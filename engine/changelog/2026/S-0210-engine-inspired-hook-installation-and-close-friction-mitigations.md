## Notes

- **Critical re-diagnosis surfaced mid-plan.** Paideia's pre-commit hook ALREADY does `REPO_ROOT="$(git rev-parse --show-toplevel)"; cd "$REPO_ROOT"` at line 32, so `validate.py` invocations DO hit the worktree's own version. The S-0209 symlink-staleness lesson (drawer `1e3071aa`) narrowed dramatically: only BASH CONTENT in the hook itself is symlink-staleness-vulnerable, not validate.py logic. This shifted the meaningful axis from "symlink vs direct script" to "how much logic lives in shared-symlinked bash vs worktree-resolved validator." Both `core.hooksPath` AND Engine's install_git_hooks.sh pattern fix the bash-content-staleness; the larger fix (thin-pre-commit / move logic to validate.py) addresses the structural property and is deferred per user scope. Captured in pushback drawer `d52f8818`.
- **Engine had been bitten by the same bug.** Engine's `install_git_hooks.sh` carries the comment *"REPO_ROOT MUST be resolved at runtime so the hook works from any worktree, not just the one that installed it (regression fix per S-0014: baking install-time REPO_ROOT broke every commit after S-0013's worktree was deleted)."* Validates the architectural direction; we're catching up to a pattern Engine empirically tested.
- **Empirical verification of relative `core.hooksPath` resolution.** In-session test: set `git -c core.hooksPath=engine/tools/hooks_test_path` with a hook at the worktree-root-relative location, made commits from BOTH worktree root AND a subdirectory (`engine/session/`). Both invocations resolved the hook to `engine/tools/hooks_test_path/pre-commit` (worktree-root-relative), confirming Git 2.50 (Apple Git-155) treats relative core.hooksPath as worktree-root-relative. Documented as Premise 1 in ADR 0100 with explicit falsifier.
- **ADR 0099 observation window confound acknowledged.** ADR 0099's acceptance criterion 4 required 3 clean closes; ADR 0100 explicitly redefines this to "3 clean closes against the combined ADR 0099 + ADR 0100 apparatus" — S-0210 = sample 1 of revised criterion. Per-failure attribution comes from pattern-matching the symptom against items, not from layered deployment.
- **First-exercise readiness gate deliberately skipped.** ADR 0053 criterion #4 fires (8 surface Consequences span), but no first-exercise readiness note is authored per the same reasoning as [ADR 0089](../../adr/0089-skill-layer1-parity-validator-check.md) and [ADR 0099](../../adr/0099-session-close-friction-mitigations.md): all four items are dogfooded in this authoring session itself. Item 1 exercised by every commit AFTER the migration; Items 2 + 4 by pytest fixtures (22 + 19 tests); Item 3 by direct hook invocation; Item 4 also live-exercised by this session's own diagnose() output (`CURRENT_ONLY` mid-session before archive write).
- **`recover_partial_close.py` failure-shape gap.** The 7-state model covers everything currently anticipated, but if a future close-failure surfaces a shape not in the enumeration, the tool emits `UNKNOWN` with "please file an Issue with the shape signature so this tool can be extended" guidance — explicit refusal beats silent guessing. Pre-S-0210, recovery from any close-time failure was bespoke + classifier-friction-bound.
