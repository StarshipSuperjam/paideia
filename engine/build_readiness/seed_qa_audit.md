## Context

The Phase 5 seed graph is built — **380 nodes / 533 edges** (516 `pedagogical_prerequisite` + 17 `historical_influence`). Phase 6 (the self-correction pipeline — tension log + Opus batch review per [ADR 0014](../../product/adr/0014-sonnet-teaches-opus-reviews.md)) is unblocked as of S-0152 and needs **real defect-rate data** to calibrate against.

The Phase 5 production audit (S-0082 → S-0122, findings at [`phase_5_production_audit_findings.md`](phase_5_production_audit_findings.md)) already SEP-anchored-reviewed ~445 graph elements and produced a **13% overall edge defect rate**. But that audit was **edge-direction-focused** — it answered "is this a genuine prerequisite?" and explicitly anti-scoped learner-facing content quality.

This census fills the gap. It checks **every** `pedagogical_prerequisite` edge and **every** node exactly once, scoring three criteria:

- **C1 — prerequisite soundness** (edges): re-validates the production audit's question as an independent drift check against the 13% baseline.
- **C2 — teaching_notes traction** (nodes): does `nodes.teaching_notes` give a learner a genuine foothold? *(net-new — the production audit never scored this)*
- **C3 — summary cold-readability** (nodes): does `nodes.summary` read clearly with zero prior context? *(net-new)*

## Scope and goal

A **chunked full census** of the seed graph, 19 routine sessions + 1 closeout. Each session scores one shard (~27 edges + 20 nodes). Output: per-criterion error rates with per-subdomain breakdown, feeding Phase 6 self-correction.

Census population (per [`seed_qa_evidence/shards.json`](seed_qa_evidence/shards.json), generated by [`engine/tools/seed_qa_shard.py`](../tools/seed_qa_shard.py)):

- **Edges:** 516 `pedagogical_prerequisite` edges. The 17 `historical_influence` edges are **out of scope** — C1 ("is this a genuine pedagogical prerequisite?") does not apply to a deliberately distinct predicate per [`PREDICATE_MANIFEST.md`](../../product/seed-graph/migrations/PREDICATE_MANIFEST.md).
- **Nodes:** all 380 nodes, every `status`. The evidence session sees each node's `status` and notes it.

Why a census, not a sample: the population is small, and at 1 routine fire/hour a full census is under a day of wall-clock. Sampling without replacement across 19 shards yields the **true** error rate with zero sampling uncertainty, for barely more budget than a loose estimate.

## Anti-scope

- **Disposition.** Evidence sessions produce candidate findings only — no Issue filing, no ADR drafts, no migration edits, no validator soft-warn proposals. Disposition is a follow-up interactive session's responsibility (matches the production audit's anti-scope).
- **Migration changes.** Evidence sessions do not edit `product/seed-graph/migrations/`. Findings about edges or nodes are recorded in evidence files; corrections (if any) are authored in build sessions after the closeout.
- **Rigor calibration.** That remains Phase 6 (the `missing_rigor_score` soft-warn's owner).
- **The closeout's synthesis.** The `seed_qa_findings.md` report and any cross-subdomain pattern analysis land at the closeout (SQA-20), not during routine evidence gathering.

## Census design — deterministic pre-sharding

Sampling without replacement across *unattended* routine sessions has no shared mutable state a session can safely append to. The solution: **pre-shard once at setup time**, embedding full element content into a committed manifest.

[`engine/tools/seed_qa_shard.py`](../tools/seed_qa_shard.py) was run once interactively at S-0156 (the Production Reads gate per [ADR 0055](../adr/0055-apply-migration-wrapping-against-production-reads-gate.md) governs routine sessions, not interactive ones). It read the live graph, deterministically shuffled (fixed seed), split into 19 shards, and wrote [`seed_qa_evidence/shards.json`](seed_qa_evidence/shards.json) with **full content embedded per shard** — each edge carries its endpoint labels + domains; each node carries `summary`, `teaching_notes`, `confidence_level`, `status`.

**Routine sessions never touch the DB.** They read their shard from the committed JSON. This sidesteps the Production Reads gate entirely.

## Per-session procedure (routine evidence session)

1. Routine-mode boot; pick the next `pending` `SQA-NN` task from `auto_target.json`.
2. Read [`seed_qa_evidence/shards.json`](seed_qa_evidence/shards.json); extract `shards["shard_NN"]` (the 2-digit zero-padded key matching the task — `SQA-01` → `shard_01`).
3. Score every edge in the shard against C1 and every node against C2 + C3, per the rubric below.
4. Write `engine/build_readiness/seed_qa_evidence/shard_NN.md` (the evidence file — scope-locked to that one path).
5. Close per the routine-mode shutdown sequence.

## Scoring rubric (fixed — cross-session consistency depends on this)

The rubric is **pinned**. 19 sessions score against it; deviation makes the aggregate rate meaningless. If a session believes the rubric is wrong, it records the concern in its `outcome_summary` and scores against the rubric anyway — the closeout adjudicates rubric concerns, not the evidence session.

### C1 — edge prerequisite soundness

For each `pedagogical_prerequisite` edge, assign one verdict. The taxonomy is the production audit's, narrowed to four — so the resulting rate is **apples-to-apples comparable** to the audit's 13%:

- **`Sound`** — the pedagogical dependency direction matches: a learner genuinely needs `source` before `target`.
- **`Defensible`** — not the canonical framing, but supportable on an alternative reading. Not a defect.
- **`Reversed`** — the pedagogical direction runs opposite the edge (`target` is actually the prerequisite of `source`, or they are co-equal).
- **`Weak-redundant`** — direction is correct, but a more proximate prerequisite already exists in the graph; this edge is a long-distance shortcut that adds little.

**Defect = `Reversed` or `Weak-redundant`** (this matches how the production audit counted "substantive defects").

Method:

- **Parametric judgment first.** Reason from the conceptual dependency directly. The default is a parametric verdict — this is a spot-check census, not a full re-audit.
- **SEP fetch is optional**, reserved for genuinely-uncertain verdicts. The empirical-fortification branch per [ADR 0059](../adr/0059-audit-time-structural-reference-fetching.md) via [`engine/tools/fetch_structural_reference.py`](../tools/fetch_structural_reference.py) is permitted for routine sessions (its first-exercise gate closed at the S-0122 production-audit closeout). Use it only when a parametric verdict cannot be reached with confidence — not as a default step.
- **Glance at `evidence`.** Note when an edge's `evidence` text is absent or does not justify the edge. This is a sub-observation recorded in the finding's notes, not a separate verdict.
- **Cross-reference the audit follow-up migrations before flagging a defect.** Migrations `0061`–`0065` (the S-0122 production-audit follow-ups) deliberately retyped, direction-flipped, pruned, or added edges. Before recording `Reversed` or `Weak-redundant`, check whether the edge was touched by one of those migrations. If a `0062` direction-flip produced the current direction and you would flag it `Reversed`, that is a genuine signal — but record it as `Reversed (contradicts migration 00NN)` so the closeout sees it is re-opening an audit decision, not finding a fresh authoring defect. This follows the S-0155 pattern: *when a topology finding traces to a deliberate audit-driven migration, name the migration rather than silently overriding it.*

### C2 — node teaching_notes traction

Yes/no. Does `teaching_notes` give a learner a genuine foothold — an intuition, a worked angle, a concrete way in? A note that merely restates the `summary`, or is abstract throat-clearing with no traction, scores **no**.

`teaching_notes` NULL or empty → automatic **no**, flagged as `teaching_notes: ABSENT`. The count of nodes with no `teaching_notes` at all is itself a finding for Phase 6.

### C3 — node summary cold-readability

Yes/no. Does `summary` read clearly to someone with **zero prior context**? Score **no** when the summary is:

- **Circular** — "X is the theory that studies X"; defines the concept in terms of itself.
- **Jargon-gated** — depends on undefined technical terms a cold reader would not know.
- **Assumes-the-concept** — only makes sense if you already understand what it is describing.

A summary can be technically accurate and still fail C3 if a newcomer cannot parse it.

## Closeout (SQA-20)

The closeout is a routine task. It reads all 19 shard evidence files and aggregates into **`engine/build_readiness/seed_qa_findings.md`**:

- Overall C1 / C2 / C3 error rates across the full census.
- Per-subdomain breakdown — every edge and node carries its `domain`, so the rates can be cut by subdomain.
- **C1 drift comparison** — the census C1 defect rate against the production audit's 13% baseline. A large divergence in either direction is a finding.
- Pattern clusters — the dominant failure shapes (e.g., reversal clusters, jargon-gated summaries in a particular subdomain).
- A **"for Phase 6 self-correction"** section — what the rates and patterns imply for the self-correction pipeline's calibration.

The closeout does **aggregation only**. Disposition — filing Issues, authoring ADR memos, proposing migration corrections — is a follow-up interactive session, matching the production audit's anti-scope.

## Known risks

- **Reviewer drift across 19 sessions.** The rubric is pinned precisely to limit this. The closeout watches for per-shard rate outliers that may indicate a session scored against a drifted interpretation rather than a real subdomain difference.
- **C1 re-opening audit decisions.** The `AUDIT-TOUCHED` field exists so the closeout can distinguish "fresh authoring defect" from "re-opens a deliberate S-0122 audit migration." Both are signal; they are different findings.
- **C2/C3 are inherently judgment calls.** Yes/no on "traction" and "cold-readability" has no SEP anchor. The pinned definitions (restates-summary → no; circular / jargon-gated / assumes-the-concept → no) are the calibration; sessions apply them literally and record rationale so the closeout can spot-check consistency.
