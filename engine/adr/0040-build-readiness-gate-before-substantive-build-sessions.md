# ADR 0040 — Build-readiness gate before substantive build sessions

- **Status:** Accepted
- **Date:** 2026-05-02
- **Deciders:** S-0027 (build-readiness gate authoring; structural halting discipline before Phase 3 first SQL migration)

## Context

The project entered the boundary between *project setup* and *project build* at S-0027. Phases 0, 1, 1.5, 2, plus the partition migration (S-0024 / S-0025) and the code-discipline contract (S-0026) all authored contracts, machinery, and operational documents — none of them committed to durable state that downstream phases must build upon. Phase 3 (SQL Schema Implementation) is the first substantive build phase: the schema is the substrate every downstream phase reads or writes to, and a subtly wrong column shape, missing CASCADE, or unconstrained enum at Phase 3 compounds across Phases 4 (graph audit), 5 (seed), 6 (self-correction), 7 (teaching), 8 (evaluation), 9 (UI). Late detection costs are higher than early authoring costs by an asymmetric margin.

The project's pre-S-0027 session model carries no halting discipline. [`session-build-lifecycle.md`](../operations/session-build-lifecycle.md)'s boot procedure reads STATE.md's "Next session work item," eager-claims a slot, and proceeds directly to authoring. Open questions and deferred decisions live in [`../../product/docs/tensions.md`](../../product/docs/tensions.md), surface during a session if they intersect the work, and resolve under build-time pressure when they do. The pattern that produced [ADR 0026](../../product/adr/0026-persistent-learner-storage-structural-not-substantive.md) — recording open questions OQ-PRIVACY-A and OQ-PRIVACY-B with "decide before Phase 3" markers, then resolving them at S-0011 with [ADR 0031](../../product/adr/0031-erasure-mechanism-and-individual-only-regime.md) before Phase 3 began — is exactly the pattern this gate generalizes. OQ-PRIVACY-A and -B were resolved in advance; T1-A through T1-E (auth model, RLS posture, JSONB enum vocabularies, SQL discipline, learner_events.context shape, surfaced in this session's adversarial analysis) were not — they would have had to resolve during S-0028's build, under pressure, with the AI's well-documented tendency to push past unresolved decisions into authoring.

The S-0027 conversation that produced this ADR named the failure mode directly: "the AI is desperate to move forward at every step. I need to bake in a stop-and-think check before each step." Routine build sessions inherit this propensity from the assistant's training; the project's standing rules ([CLAUDE.md](../../CLAUDE.md) Pushback rule, Auto-mode interrupt criteria) apply to in-session judgment but do not structurally precede a session. Halting requires a procedural surface that a build session cannot bypass.

The compound-drift mode [ADR 0038](0038-expression-contract-for-ai-authored-code.md) names for code has an analog at the session level: a build session that begins with one or more Tier 1 decisions unresolved, improvises resolutions in the moment, and produces internally coherent artifacts that pass mechanical gates (Layer 2) and survive cold review (Layer 3) — because the gates and the review check artifact-vs-contract alignment, not contract-vs-reality alignment. If the contract was authored under build pressure with a wrong premise, every layer compounds the wrong premise. The gate model ADR 0038 + ADR 0039 build *cannot* catch session-premise drift; that catch belongs to a halting discipline that runs *before* the contract is authored.

The fix is structural. Every substantive build session is preceded by a gate session whose deliverable is a build-readiness report. The gate session's job is *halting discipline* — surface red flags via adversarial analysis (Explore agents, multiple perspectives), triage findings (Tier 1 must resolve / Tier 2 settle in advance / Tier 3 name as deferred), resolve Tier 1 with the user via conversation or AskUserQuestion, author resolution artifacts (ADR amendments, vocabularies, sub-contracts), and produce a single build-readiness report that the build session reads at boot. The build session refuses to author if Tier 1 is not resolved. Halting is a feature, not a regression.

## Decision

The project authors a build-readiness gate protocol — a halting discipline that precedes every substantive build session. The operational surface lives at [`../operations/build-readiness-gate.md`](../operations/build-readiness-gate.md). This ADR is the citable contract.

The gate's posture: every substantive build session reads a build-readiness report at boot. The report lives at `engine/build_readiness/<phase>_<chunk>.md` and is authored by a preceding gate session. The build session's boot procedure (per [`../operations/session-build-lifecycle.md`](../operations/session-build-lifecycle.md), amended in this commit) checks for the report; if absent, the session converts to a gate session; if present with unresolved Tier 1 findings, the session halts and escalates to the user. The discipline is structural — the build session has no path to substantive authoring that bypasses the readiness check.

The gate's scope governs *substantive build sessions*: sessions that author or modify load-bearing artifacts that downstream phases depend on. Concretely: Phase 3 (SQL schema), Phase 4 (graph audit code that the schema is fed to), Phase 5 (seed graph content), Phase 6 (self-correction code), Phase 7 (teaching layer), Phase 8 (evaluation harness), Phase 9 (UI prototype), and any future phases that produce schema, durable code, or content that subsequent phases consume. *Operational sessions* — health checks per [ADR 0022](0022-periodic-project-health-checks.md), small refinements to operational docs, ENGINE_LOG-only edits, retrievability cleanups — do not require a gate session; their modification surface is already audited at shutdown. The gate session itself is operational by this definition; gate sessions are not preceded by gate sessions (no infinite regression).

The gate's deliverable is a build-readiness report with three sections: **Pre-session decisions** (Tier 1 resolutions with author and rationale), **Tier 2 decisions** (concrete answers to questions the build session would otherwise improvise: column types, constraints, defaults, indexes, schema shapes), **Tier 3 forward pointers** (decisions explicitly deferred with named decide-before phases). The report ends with success criteria the build session inherits from the build_plan chunk plus any session-specific verification steps the gate exercise surfaced. Format and worked example live in [`../operations/build-readiness-gate.md`](../operations/build-readiness-gate.md).

The gate's amendment discipline matches the structural pattern of [ADR 0036](0036-expression-contract-for-inward-documents.md), [ADR 0038](0038-expression-contract-for-ai-authored-code.md), and [ADR 0039](0039-universal-expression-contract-across-ai-authoring-patterns.md). Refinements within the gate model are cheap because they preserve the discipline; restructuring the gate model itself is expensive because it is what the discipline binds. Refining the build-readiness report template, adding Explore-agent perspectives, sharpening the triage criteria, adjusting which session types are "substantive" — all ENGINE_LOG-tracked refinements. Removing the gate, making the report optional, dropping the "build session refuses to author with unresolved Tier 1" rule — posture changes requiring superseding ADR.

## Consequences

- **The operational document is the working surface; this ADR is the citable contract.** [`../operations/build-readiness-gate.md`](../operations/build-readiness-gate.md) carries the gate-exercise procedure, the build-readiness report template, the worked example (Phase 3 SQL gate exercise authored in this same session), and the Explore-agent prompts. Refinements land there under ENGINE_LOG tracking; posture changes require superseding this ADR. The same separation [ADR 0036](0036-expression-contract-for-inward-documents.md), [ADR 0038](0038-expression-contract-for-ai-authored-code.md), and [ADR 0039](0039-universal-expression-contract-across-ai-authoring-patterns.md) maintain between themselves and their operational surfaces.

- **Phase 3 (S-0028) opens with a complete build-readiness report.** S-0027 produces [`../build_readiness/phase_3_sql.md`](../build_readiness/phase_3_sql.md) as the gate's first exercise. T1-A through T1-E (the five Tier 1 findings from the adversarial analysis preceding S-0027) are resolved in advance. T2-A through T2-H (column shapes, constraint forms, FK targets) are documented as concrete decisions S-0028 implements without re-deciding. T3-A through T3-G are named as deferred forward-pointers. S-0028's boot reads the report; if it surfaces Tier 1 items not resolved, S-0028 halts and escalates.

- **The gate consumes session capacity by design.** A phase that surfaces deep Tier 1 issues may consume multiple gate sessions before its build session opens. This is the intended cost: the alternative — building under unresolved decisions and retrofitting later — is more expensive in compound-drift terms. The session register (`engine/session/register_state.json`) does not distinguish gate sessions from build sessions; both claim slots through the existing eager-claim ritual. STATE.md's `Last build session` row uses the same shape for both; the distinction surfaces in the session's `working_on` field.

- **Auto-mode interrupt criteria invert for gate sessions.** [CLAUDE.md](../../CLAUDE.md)'s standing rules name escalation criteria for build sessions running autonomously: irreversible-with-unclear-path, unsolvable, destructive-action-confirmation. Gate sessions are *conversational by default* — their job is to surface decisions and resolve them with the user, not to push through autonomously. A gate session that finds itself wanting to "auto-resolve" a Tier 1 issue rather than escalate has misidentified its mode. The gate's deliverable is co-authored with the user; the AI surfaces, the user directs, the AI authors the resolution artifacts. The build-readiness report is the contract the build session inherits.

- **The first exercise is dogfooding.** The S-0027 gate exercise on Phase 3 SQL is also the procedure's first execution. Friction surfaces in the procedure itself — Explore-agent prompt redundancy, triage-tier ambiguities, report-section overlap — are refinement signals, ENGINE_LOG-tracked. The mechanism is calibrated to the session-level compound-drift mode; if a different failure mode dominates, calibration changes.

- **The gate plays well with multi-session phases.** A phase that requires multiple build sessions (Phase 5's seed graph build is the canonical example) opens with a single gate session that produces one report, then runs N build sessions that all read the same report. Per-session decisions internal to the build sessions (e.g., "which subdomain do we seed first") are not Tier 1; the gate addresses cross-session decisions (column shapes, file layout, validation contracts).

- **Refinement under load is expected.** The first non-Phase-3 gate exercise (likely S-0029 / Phase 4) is the second real test. The triage criteria (what counts as Tier 1 vs Tier 2 vs Tier 3) will sharpen with use. The build-readiness report template will refine as authors discover which sections are load-bearing and which are decorative. These are refinement signals — ENGINE_LOG-tracked — not contract failures. If the gate produces friction disproportionate to the compound-drift it prevents, the contract gets revised under its own amendment discipline.

- **No grandfathering for prior phases.** The gate applies to every substantive build session from S-0028 forward. Pre-S-0028 phases authored under no gate (Phase 0, 1, 1.5, 2 plus the partition migration) do not retroactively need build-readiness reports. The contract takes force at acceptance.

- **The gate consumes the [ADR 0039](0039-universal-expression-contract-across-ai-authoring-patterns.md) "no row, no authoring" check.** When a substantive build session opens a new authoring pattern (e.g., Phase 7 lands the first TypeScript artifact), the gate session is where the pattern's row in [`../operations/expression-contract-instantiation.md`](../operations/expression-contract-instantiation.md) gets authored. The build session inherits a complete row; "no row, no authoring" is enforced upstream of the build session's boot.

- **The "Last build session" table row pattern in STATE.md continues unchanged.** Gate sessions and build sessions both update STATE.md's `Last build session` and `Prior build session` rows at shutdown. The session-type distinction is recoverable from `working_on` summaries and from the session's outcome_summary, both of which are linked from the row. No new STATE.md schema is needed; the gate is a session-type, not a session-class.
