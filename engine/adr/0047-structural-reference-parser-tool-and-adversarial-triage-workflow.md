# ADR 0047 — Structural-reference parser tool and adversarial-triage workflow

- **Status:** Accepted
- **Date:** 2026-05-03
- **Deciders:** S-0039

## Context

[ADR 0046](../../product/adr/0046-structural-reference-posture-extends-to-philosophy-reference-works.md) settles the *posture* for consuming a class of curated philosophy reference works (SEP, IEP, Routledge Encyclopedia of Philosophy, Oxford Reference, Wikipedia) as structural priors for Phase 5 seed authoring: graph-shape value (cross-reference adjacency, concept inventory, entry-presupposition signals) is distinct from content/prose value, the structural extraction is fact-shaped per Feist, and no source prose is hosted, redistributed, or surfaced to learners. ADR 0046 does not name an *operational mechanism* for that consumption.

The naive mechanism — uploading source PDFs into AI context — is unworkable. Routledge alone is ~10K pages; Oxford Reference and SEP combined are larger still. Beyond the token-count infeasibility, uploading the source defeats the structural-extraction-only posture in ADR 0046 (the AI consumes the prose along with the structure) and stresses [ADR 0011](../../product/adr/0011-no-hosted-copyrighted-material.md)'s no-hosted-copyrighted-material commitment operationally (full-source uploads to AI context cross the spirit of "no hosting" even if the technical letter is preserved).

A second naive mechanism — letting Claude crawl the source live via WebFetch on every Phase 5 sub-session — is also unworkable. Per-session network traffic is unreliable, throughput-bounded, and burns context budget on raw markup the AI must then re-extract structure from. The cost is paid every session against the same source; the structure does not change between sessions.

The replacement workflow uses a local Python parser to extract graph-shape signal (entry inventory + cross-reference adjacency + section/topic tags + per-entry word-counts and URLs) from any source the user has acquired through legitimate channels (free public access for SEP / IEP / Wikipedia; institutional library access or subscription for Routledge / Oxford). The parser produces a small structured JSON *focusing brief*; the brief becomes the upload-able artifact; the LLM consumes the brief for adversarial triage against current graph state and parametric knowledge of the concepts.

The parser is decoupled from Phase 5 — a standing engine tool runnable ad-hoc whenever a source is acquired, not a phase deliverable that gates Phase 5 sub-sessions. Phase 5 epistemology (and every other Phase 5 sub-session) can proceed without the parser; the parser augments whatever is already in the graph whenever it gets run. The decoupling is what justifies landing the parser tooling at S-0039 (the slot the Phase 5 epistemology gate had been projected to take): it slips the gate by exactly one session and lands the operational mechanism before any Phase 5 sub-session needs to consume it.

Four design choices warrant ADR-level commitment because each propagates to multiple consuming sessions and each has substantive alternatives the project might want to revisit:

1. The parser's lifecycle and coupling to phases.
2. The parser's per-source-vs-per-document-type architecture.
3. Whether parser output is persisted across runs (and whether a cross-source corroboration mechanism is built).
4. How the LLM consumes parser output — defaulting to acceptance vs defaulting to rejection.

## Decision

**The parser is a standing engine tool, not a phase-coupled deliverable.** It lives at `engine/tools/parse_structural_reference.py` (single-file by default; promoted to a `engine/tools/parser/` subpackage only if size and module separation genuinely warrant it). Its lifetime equals the project's lifetime. It is runnable ad-hoc by the user whenever a structural-reference source is acquired. No Phase 5 sub-session is gated on the parser's output; the parser augments whatever is already in the graph whenever it runs. ROADMAP.md is unchanged; the parser does not occupy a phase slot.

**The parser is document-agnostic with per-document-type heuristic configs.** One parser, multiple input adapters (HTML now; PDF and plaintext later), all sharing a common structural-detector layer. Per-document-type configs are small declarative dicts (heading-detection rules, cross-reference patterns, entry-boundary heuristics) at module top — `ENCYCLOPEDIA_CONFIG`, `DICTIONARY_CONFIG`, `TEXTBOOK_CONFIG`, etc. The configs name *document type* (encyclopedia, dictionary, textbook), not *source* (SEP, Routledge); a per-source parser would require source-specific code, which would require the user to upload source samples to the AI to write the rules — defeating the workflow's point. The HTML input adapter implemented at S-0039 handles SEP-shaped HTML specifically; the PDF input adapter is named here as planned future work and lands when the user has acquired a PDF source and wants to run the parser against it. Source acquisition remains the user's responsibility through legitimate channels per ADR 0046.

**Output is signal, not truth; ephemeral, not persisted.** The parser is recall-prioritized over precision: a parser that finds 60% of real cross-references with some noise is useful; a parser that finds 100% of cross-references but is brittle to source-format quirks is not. The LLM's parametric knowledge handles the noise — a parser-surfaced candidate the LLM does not recognize anywhere gets rejected, whether the cause is genuine niche-ness (concept too niche for generally-accepted-domain-knowledge, does not belong in Paideia anyway), a parser bug, or a source-material idiosyncrasy. Same outcome in all three cases, all correct rejections. The parametric-knowledge boundary is the right credibility threshold for the kind of pedagogically-prerequisite-shaped education Paideia does. Parser output is a JSON focusing brief consumed by an adversarial-triage prompt and then **discarded**. There is no `engine/parser_findings/` directory, no cross-source corroboration aggregator, no aging policy. The graph is the only durable artifact; accepted candidates land as `INTERPRETED` nodes per [ADR 0030](../../product/adr/0030-confidence-level-evidentiary-mode-on-nodes.md). The rationale for non-persistence: persistence accumulates noise across runs, and "candidate appears in N runs" is a false credibility signal when same-source re-runs are indistinguishable from independent corroboration without bookkeeping the user must actively maintain. The reversibility of this choice is preserved — a follow-up ADR can add a corroboration mechanism without invalidating this ADR's other decisions.

**Triage is adversarial; the LLM defaults to rejection.** The triage prompt template (deferred in this ADR; lands at first-real-use when there is actual SEP output to triage against actual graph state) frames the LLM as an adversarial reviewer of parser candidates. Every candidate must pass three rejection filters before survival: (1) candidates not recognized in parametric knowledge are rejected (parametric-knowledge boundary as the credibility threshold); (2) candidates that duplicate-name or differently-name existing graph nodes are rejected (the parser's recall-prioritization will surface known concepts under variant spellings; the LLM resolves variants against current graph state); (3) candidates whose semantic content is out-of-scope for Paideia's pedagogical-prerequisite framing per [ADR 0001](../../product/adr/0001-pedagogical-edges-not-historical.md) are rejected (the parser surfaces historical-influence and editorial-curation concepts; not all of those are pedagogical prerequisites). Survivors require explicit positive justification per candidate naming why the concept belongs and what its prerequisite shape is. The triage prompt's exact wording is deferred to first-real-use because authoring it without a real focusing-brief sample to test against would be premise-drift bait — the prompt's effectiveness depends on what the JSON actually carries when run against a real entry, which only the implementation reveals.

## Consequences

**[ADR 0046](../../product/adr/0046-structural-reference-posture-extends-to-philosophy-reference-works.md) is unchanged.** This ADR is the *operational mechanism* for the posture ADR 0046 establishes; it does not revise the posture. ADR 0046's class boundary (SEP, IEP, Routledge, Oxford Reference, Wikipedia plus future admissions per ADR), source-acquisition discipline, fact-shaped extraction principle, and Generative Graph Independence preservation all stand exactly as written.

**[ADR 0011](../../product/adr/0011-no-hosted-copyrighted-material.md) is unchanged.** The parser extracts fact-shaped graph-shape signal (concept names, cross-reference adjacency, section paths, word counts, URLs); no source prose is hosted, redistributed, or surfaced to learners. The focusing-brief JSON is consumed by an in-process LLM triage and then discarded; it does not flow into the graph as content. Test fixtures cached at `engine/tools/test_fixtures/sep/<entry-slug>.html` are parser-implementation-internal, never reach the production graph or any user-facing surface, and exist only as test inputs for the detectors and adapter — their use is bounded by what test fixtures normatively are. The fixtures are SEP HTML acquired under SEP's explicit crawl-for-indexing permission per [`product/docs/content-strategy.md`](../../product/docs/content-strategy.md) Tier 4.

**The pattern is portable to future-domain expansions.** Theology, history, literature, law, and other domain expansions per [`product/docs/expansion.md`](../../product/docs/expansion.md) reuse the same parser plus the same triage workflow. New input adapters (PDF for Routledge, plaintext for plain-text encyclopedias) and new per-document-type configs (dictionary, textbook, primary-source-anthology) extend the parser without invalidating its architecture. The shared structural-detector layer is what carries the portability; per-document-type configs absorb format variation; per-source idiosyncrasies are absorbed by the LLM's adversarial triage rather than parser-level branching.

**The no-corroboration choice is reversible.** A future need — say, surfacing concepts that appear across three independent reference works as higher-credibility candidates — can land as a follow-up ADR adding a corroboration mechanism (a small persistent index keyed by candidate-id, updated per parser run). That follow-up does not invalidate this ADR's other decisions; the lifecycle, document-agnostic, and adversarial-triage choices remain in force. The reversibility is named here so a future session reading this ADR knows the exclusion is a deliberate tradeoff, not an oversight.

**Triage prompt template lands at first-real-use.** The prompt's exact wording is a deferred deliverable, not an ADR omission. The first session that runs the parser against a real SEP entry (or any other source) authors the triage prompt against the real focusing-brief output, captures the prompt in `engine/operations/structural-reference-triage.md` (or a similarly-named ops doc), and lands a one-line ENGINE_LOG `Added` entry. The triage prompt is documented as a template with placeholder fields the consuming session fills in; it is not a per-source prompt.

**PDF input adapter lands at first-PDF-source-acquisition.** The HTML adapter implemented at S-0039 is sufficient for SEP, IEP, and Wikipedia (all HTML-native or HTML-renderable). PDF adapter implementation defers to the session that has acquired Routledge or Oxford Reference and wants to run the parser. The deferral is concrete: the adapter interface is in place, the structural-detector layer is in place, the per-document-type configs are in place; the PDF adapter is one new file (likely using `pypdf` or `pdfplumber`) that produces the same `AdapterOutput` the HTML adapter produces. No architectural change is required.

**PDF adapter landed at S-0096 per Issue #34** (Consequences amendment). The deferral closed when the user staged Routledge / Oxford / Cambridge PDFs at `~/Documents/Claude_Files/temp/` for build-session iteration. The implementation honors decision 1's source-agnostic posture more strictly than the HTML adapter does — `HTMLInputAdapter` carries SEP- and IEP-specific URL patterns by accident of HTML's source-specific anchor conventions, while `PDFInputAdapter` carries no per-source code branches. Multi-fallback at every detector: title from `/Title` metadata then first-page-largest-font; heading from pypdfium2 outline tree (non-recursive — pdfminer's recursive `get_outlines` blows the default 1000-deep recursion limit on encyclopedia-scale outlines like Routledge's 2041 entries) then font-size buckets sampled and emitted within `PDF_FONT_EMIT_PAGE_CAP`, then a `Chapter|Section|Part|Article` text-pattern with ≤8-word + ≤80-char + prose-connector-reject filters; anchors from link annotations PLUS textual editorial markers (`See:`, `See also:`, `(see X)`, `→`, `Cf.`) — both paths emit canonical `internal:<slug>` hrefs picked up by one new `ENCYCLOPEDIA_CONFIG["cross_reference_patterns"]` entry. Bulk text extraction via pypdfium2 (~30× faster than pdfplumber's pdfminer backend at encyclopedia scale). Image-only PDFs raise `ValueError` loudly — OCR is the user's prep responsibility. Paragraph and list-item element kinds are NOT collected; no downstream detector consumes them.

The same session landed the multi-entry partitioning extension to `extract_entries()` per its docstring's "future" note: `partition_at_entry_boundaries()` splits the elements list at level-1 heading boundaries; `extract_entry_from_partition()` runs the existing detector pipeline against a scoped element list. Single-`<h1>` HTML still yields exactly one Entry (the 6 SEP fixture integration tests at `TestSEPFixtureIntegration` pass unchanged — load-bearing regression bar). Full-volume PDFs yield N Entries. A post-emission element-reorder pass uses `full_text.find` (with monotonically advancing pointer for consecutive headings) and captured `match.start()` for anchors to interleave headings + anchors by document position; without this the heading- and anchor-emission phases write disjoint position ranges and all cross-references concentrate into the LAST entry of any multi-entry brief.

Three-source convergence at the C7 landing: Routledge's outline-driven inventory yields 2041 entries with 1177 (58%) carrying cross-references and mean confidence 0.62 — the architecture's strong case. Cambridge and Oxford remain sparse (20 and 15 entries respectively) because their outlines enumerate book sections / letter sections only and their entries are at body font size. The page-header-as-entry-indicator path that would close that gap is filed as Issue #48 — source-agnostic in shape (universal first-line-of-page detection) but architecturally distinct from the outline + font-size + text-pattern stack landed here.

Test coverage at C7: `TestPDFInputAdapter` (11 cases) exercises each fallback layer against `reportlab`-synthesized PDF fixtures (test-only dep added to `engine/tools/requirements.txt`); `TestMultiEntryPartitioning` (8 cases) exercises the partition logic directly including the per-partition cross-reference distribution regression bar. No third-party PDF prose lands in committed artifacts — the licensing posture stays clean per ADR 0011 / ADR 0046, with the three real test PDFs staying in `~/Documents/Claude_Files/temp/` (worktree-external) and never copied into the repo.

### Amendment (S-0103) — source-identity anonymization and body-font-entry/page-header-runner detector tier

Two paired changes landed at S-0103, one bug fix and one feature, both targeting the same parser file in one combined session per user direction. They share a load-bearing structural primitive (the publication-name pattern family) introduced by the bug fix and consumed by the feature.

**Source-identity anonymization (Issue #49)** preserves Decision 4 (parametric-knowledge boundary) by stripping pre-triage source-identity surfaces from the brief. The strip operates at three layers:

  - **Serialization boundary**: `serialize_brief()` removes `source_path` (top-level) and `source_url` (per-entry) from the JSON output via a `dataclasses.asdict()` post-filter. The in-memory `FocusingBrief` and `Entry` dataclasses retain the fields because internal consumers (notably `detect_entry_id`'s URL-slug derivation) still need them.
  - **Title detection**: `strip_publication_name_suffix()` removes structurally-shaped publication-name suffixes from candidate titles (e.g., a title that ends with `<X> Encyclopedia/Dictionary/Companion/Handbook/Reference/Compendium of/to <Y>` after a separator is shortened). `is_publication_name_shaped()` rejects whole-shape candidates so the H1 fallback chain takes over.
  - **Config**: the previously-enumerated per-publisher patterns in `ENCYCLOPEDIA_CONFIG["title_suffix_strip_patterns"]` are subsumed by the structural stripper. The alternation tokens (Encyclopedia, Dictionary, Companion, Handbook, Reference, Compendium) are descriptors of reference-work types, not source-identity tokens — preserving the parametric-knowledge boundary at the project's own config layer.

**Body-font-entry/page-header-runner detector tier (Issue #48)** extends Decision 1 (source-agnostic, multi-layer fallback) with a third heading-detection tier. Order is now outline → font-size → page-header. The new method (`PDFInputAdapter._emit_page_header_headings()`) gates by `outline_count >= 50` to mirror the font-size short-circuit, then per-page extracts the first non-empty line, applies the page-header-shape predicate (short, capitalized, no terminal punctuation, distinct from body content), and splits on whitespace or case-transition boundaries.

**Cross-deliverable anonymization invariant** (load-bearing): page-header runners commonly carry the publication name on alternating pages per standard typography, so the new emit path was a potential leak vector. `is_publication_name_shaped()` is applied at three locations in defense-in-depth: the new emit method (skips publication-name-shaped lines at emit-time AND re-checks each candidate token after splitting), the partition layer in `extract_entries()` (drops partitions whose first heading is publication-name-shaped — catches leakage from any path including outline / font-size / text-pattern), and the title-detection chain (rejects publication-name candidates and continues to H1 fallback). The implementation discipline holds throughout: code identifiers, config values, test data, comments, and this ADR's amendment text use structural patterns and synthetic publisher tokens (`Acme`, `FakeReference`, `MockPublisher`, `GenericSource`) — no enumerated source-identity names. Test fixtures synthesize adversarial alternating runners (entry-pair on odd pages, publication-name on even pages); a recursive walk over the resulting brief asserts no string at any depth matches the structural patterns.

**Class-admission ADR per ADR 0046 deferred until graph-mining session.** ADR 0046's class-boundary supersession clause invites an admit-to-class ADR when a new structural class is exercised end-to-end against a real source for graph mining. The body-font-entry/page-header-runner class is recognized by the parser at S-0103 but not yet exercised against a real source for downstream graph mining; the admit-to-class ADR lands when that consumption occurs.

**Per [ADR 0036](0036-expression-contract-for-inward-documents.md) amendment asymmetry**: the original Consequences claims (no-corroboration choice, source-agnostic posture, multi-fallback ladder) are preserved as historical record; this amendment supersedes operational claims about source-identity surfaces in the brief output (the brief now anonymizes; the original landing left source identity present) and extends the operational claim about the multi-fallback ladder (now three tiers, not two).

**Test coverage at S-0103**: `TestSerializeBriefAnonymization` (5 tests), `TestPublicationNamePattern` (10 tests + parametrize expansions to 32 test variants total), `TestPageHeaderEntryIndicator` (12 tests), `TestPageHeaderAnonymizationInvariant` (6 tests, the cross-deliverable invariant gate). Existing `TestSEPFixtureIntegration` regression bar passes unchanged — the structural stripper subsumes the previous SEP-style enumerated suffix structurally.

**The parser package falls under the existing Python/engine pattern row in [`engine/operations/expression-contract-instantiation.md`](../operations/expression-contract-instantiation.md).** No new pattern row is added. Layer 1 source-of-truth is [`engine/operations/code-discipline.md`](../operations/code-discipline.md); Layer 2 is the existing ruff + mypy --strict + pytest gate stack invoked from `validate.py`'s `validate_code_gates()`; Layer 3 is the cold-context review pass at session shutdown. The parser package's module-level contract block (per Layer 1) names purpose, dependencies, the no-persistence rule, the recall-over-precision orientation, and the LLM-triage handoff — fully auditable in prose before code is read.

**`engine/tools/requirements.txt` gains `beautifulsoup4`.** The HTML input adapter uses BeautifulSoup4 (`bs4`) for HTML parsing; this is added to `engine/tools/requirements.txt` at S-0039 alongside the parser implementation. The PDF adapter at S-0096 added `pdfplumber>=0.11.0` (structural metadata), `pypdfium2>=4.0.0` (bulk text extraction; pinned explicitly even though pdfplumber re-exposes it transitively), and `reportlab>=4.0.0` (test-only — `TestPDFInputAdapter` synthesizes PDF fixtures in-test rather than committing third-party PDF blobs).

**`ROADMAP.md` is unchanged.** The parser is decoupled from phases per the lifecycle decision above; it lives in `engine/tools/` and exists outside the phase sequence. ENGINE_LOG `[Unreleased]` `Added` carries the dated record.

**Open future ADRs.** None forced. Likely future ADRs include: a follow-up if cross-source corroboration becomes warranted (per the reversibility note); a small operational ADR documenting the triage prompt template once it lands; a per-source acquisition ADR if a paywalled member of the ADR 0046 class is acquired for the first time. None of these are blocked on this ADR.
