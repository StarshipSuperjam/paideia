# ADR 0039 — Universal expression contract across AI authoring patterns

- **Status:** Accepted
- **Date:** 2026-05-02
- **Deciders:** S-0027 (build-readiness gate authoring; pre-Phase-3 widening of the code contract from Python-only to all AI authoring patterns)

## Context

The project carries three expression contracts, each of which constrains how the AI expresses itself for a single surface. [ADR 0027](../../product/adr/0027-rendering-policy-prompt-layer-contract.md) governs learner-facing prose through a forbidden-token enumeration; the surface ships verbatim as Sonnet's system prompt at Phase 7. [ADR 0036](0036-expression-contract-for-inward-documents.md) governs inward-facing project documentation through a positive voice characterization audited at authoring time; the surface is the present-state artifact's purpose. [ADR 0038](0038-expression-contract-for-ai-authored-code.md) governs Python under [`../`](../) through three discipline layers (contract-first prose, mechanical gates, cold-context review); the surface compensates for the weak human-audit channel that code presents at scale.

Each contract is calibrated to its surface — what it protects, what audit channel exists, where drift originates. The three sit as siblings.

The wall the three contracts erect is not coextensive with the project's AI authoring surface. ADR 0038 explicitly scopes itself to "all `*.py` files under [`../`](../)" and names that "product code under [`../../product/`](../../product/) when it lands… is governed by the same three-layer posture; the language-specific gate stack adapts at the session that lands the first product code." That deferral was correct at S-0026 — Python under engine/ was the only code surface — but the deferral is also load-bearing: future authoring patterns proceed unguarded by default until each one's contract lands.

The patterns currently outside the wall are concrete and adjacent. SQL migrations land at [Phase 3](../../build_plan/P_1_sql_schema.md) (S-0028 next); the schema is the substrate every downstream phase reads or writes to. TypeScript edge functions land at [Phase 7](../../build_plan/P_7_teaching_layer.md) per [`../../product/docs/architecture.md`](../../product/docs/architecture.md). YAML configuration (Supabase project config, future CI), shell scripts beyond the existing `pre-commit` and short wrappers, learner-facing UI text (per the open OQ-OUTWARD-VOICE in [`../../product/docs/tensions.md`](../../product/docs/tensions.md)) all currently sit outside any of the three contracts. The pattern of adding a contract per surface as each surface arrives — the path that produced ADR 0027 → ADR 0036 → ADR 0038 — leaves a gap between when an authoring pattern's first artifact lands and when its contract takes force.

The compound-drift mode [ADR 0038](0038-expression-contract-for-ai-authored-code.md) names is not Python-specific. SQL admits the same failure shape: a wrong premise about Postgres CASCADE semantics, a wrong premise about JSONB constraint behavior, a wrong premise about RLS policy evaluation built upon consistently across many migration lines, producing internally coherent SQL that passes ad-hoc inspection. TypeScript admits it. Shell admits it. Even the prose contracts admit a softer version — the rendering policy's forbidden-token enumeration is calibrated against drift in the form of phrases the AI might emit consistently across many turns. What differs across patterns is which layers the contract needs, not whether a contract is needed.

The fix is structural. The three-layer posture from [ADR 0038](0038-expression-contract-for-ai-authored-code.md) generalizes: Layer 1 forces the auditable surface up into prose (a contract block, a positive voice description, a forbidden-token list — the form depends on the pattern), Layer 2 mechanizes what mechanization can catch (lint, type-check, schema validation — the toolchain depends on the pattern), Layer 3 introduces fresh eyes (cold-context review — applicable wherever compound drift is the dominant risk). Each AI authoring pattern instantiates the layers it needs; what each pattern must declare is *which layers it carries*, not whether to participate.

## Decision

This ADR generalizes [ADR 0038](0038-expression-contract-for-ai-authored-code.md)'s three-layer posture into a universal expression contract that binds every AI authoring pattern in the project. The existing pattern-specific contracts ([ADR 0027](../../product/adr/0027-rendering-policy-prompt-layer-contract.md), [ADR 0036](0036-expression-contract-for-inward-documents.md), [ADR 0038](0038-expression-contract-for-ai-authored-code.md)) are pre-existing instantiations of this contract; they remain in force unchanged. New AI authoring patterns must declare their layer instantiation before any artifact is authored under them.

The contract's posture: every AI authoring pattern in the project requires Layer 1 (contract-first authoring rules — prose form depends on the pattern) and at least one of Layer 2 (mechanical gates) or Layer 3 (cold-context review). Layer 1 is universal because the auditable surface forced up into prose is what every other discipline judges against; without it the other layers have nothing to compare implementation to. Layer 2 and Layer 3 are independently sufficient depending on the pattern's audit channel — patterns whose drift is mechanically detectable need Layer 2; patterns whose drift requires premise-level reading need Layer 3; patterns where both modes coexist (Python, SQL) need both.

The contract's working surface is the pattern instantiation table at [`../operations/expression-contract-instantiation.md`](../operations/expression-contract-instantiation.md). Each row names a pattern (prose/learner-facing, prose/inward, Python/engine, SQL/migrations, TS/teaching-layer, etc.), the Layer 1 source-of-truth document, the Layer 2 gate (or "N/A — Layer 1+3 only"), the Layer 3 review trigger (or "N/A — Layer 1+2 only"), and the scope path under which the pattern's artifacts live. Adding a new pattern adds a row before authoring begins; "no row, no authoring" is the discipline gate.

The contract's scope governs every AI-authored artifact in the repository. Patterns currently lacking a row are (a) any of the four already-decided rows (prose/learner-facing per [ADR 0027](../../product/adr/0027-rendering-policy-prompt-layer-contract.md), prose/inward per [ADR 0036](0036-expression-contract-for-inward-documents.md), Python/engine per [ADR 0038](0038-expression-contract-for-ai-authored-code.md), SQL/migrations per this session's [`../operations/migration-discipline.md`](../operations/migration-discipline.md)) — these are filled at this session's authoring; or (b) any pattern whose first artifact has not yet landed — these must add their row at the session that opens the pattern, not later.

The contract's amendment discipline matches the structural pattern of [ADR 0036](0036-expression-contract-for-inward-documents.md) and [ADR 0038](0038-expression-contract-for-ai-authored-code.md). Refinements within the layer model are cheap because they preserve the discipline; restructuring the layer model itself is expensive because it is what the discipline binds. Adding a row to the instantiation table for a new pattern is a refinement (ENGINE_LOG-tracked); changing a row's Layer 2 toolchain is a refinement; tightening Layer 3 review prompt templates is a refinement. Restructuring the layer count, abandoning the universal-contract framing, or removing the "Layer 1 + at least one of L2/L3" minimum are posture changes requiring superseding ADR.

## Consequences

- **The instantiation table is the working surface; this ADR is the citable contract.** [`../operations/expression-contract-instantiation.md`](../operations/expression-contract-instantiation.md) carries the per-pattern rows and the worked-example test (an authoring session reading the table cold can answer "do my changes carry the layer instantiation their pattern declares?"). Refinements land there under ENGINE_LOG tracking; posture changes require superseding this ADR. The same separation [ADR 0036](0036-expression-contract-for-inward-documents.md) and [ADR 0038](0038-expression-contract-for-ai-authored-code.md) maintain between themselves and their operational surfaces.

- **The four existing pattern contracts remain authoritative for their patterns.** [ADR 0027](../../product/adr/0027-rendering-policy-prompt-layer-contract.md) is the Layer 1 source-of-truth for learner-facing prose; this ADR does not restate or modify its content. [ADR 0036](0036-expression-contract-for-inward-documents.md) is the Layer 1 source-of-truth for inward-facing documentation; this ADR does not restate or modify it. [ADR 0038](0038-expression-contract-for-ai-authored-code.md) is the citable contract for Python under engine/; this ADR widens the universe ADR 0038 declared but did not enforce. Each pattern's row in the instantiation table cites its source-of-truth; this ADR sits above them as the universal binding.

- **SQL/migrations gains its row as part of this session.** The Layer 1 source-of-truth is [`../operations/migration-discipline.md`](../operations/migration-discipline.md) (authored in this same commit). The Layer 2 gate is the SQL-gates branch in [`../tools/hooks/pre-commit`](../tools/hooks/pre-commit) invoking `validate.py`'s `validate_sql_gates()` (also authored in this commit). The Layer 3 review trigger is the cold-review pass extension in [`../operations/session-shutdown-sequence.md`](../operations/session-shutdown-sequence.md) step 3 (also amended in this commit). The instantiation has all three layers because SQL admits both mechanical drift (constraint shape, syntax, missing CASCADE) and premise drift (compound-wrong assumptions about Postgres semantics).

- **Future authoring patterns must declare layer instantiation before opening.** When the first TypeScript artifact lands at Phase 7, the session that opens that pattern adds its instantiation row to the table before authoring the artifact. Same for YAML config, shell scripts beyond the existing wrappers, the learner-facing UI text surface that OQ-OUTWARD-VOICE names. The "no row, no authoring" discipline is what the build-readiness gate ([ADR 0040](0040-build-readiness-gate-before-substantive-build-sessions.md), authored in this same session) enforces structurally.

- **Layer 1 is universal; Layer 2/3 are pattern-specific.** Every pattern carries a Layer 1 source-of-truth — prose, code, schema, configuration. Without Layer 1 there is no auditable surface, and the other layers' compensation has nothing to bind to. Whether Layer 2 or Layer 3 (or both) applies depends on the pattern's audit channel: prose's human-readable surface makes Layer 1 + spot-check sufficient (ADR 0036); code and SQL's compound-drift exposure makes Layer 1 + Layer 2 + Layer 3 necessary; learner-facing prose's hidden-machinery exposure makes Layer 1 + Layer 3 (ADR 0027's authoring-session-and-shutdown enforcement) sufficient and Layer 2 unreliable (forbidden-token regex catches some but compound drift escapes). The instantiation table records each pattern's choice and its rationale.

- **The instantiation table itself sits under [ADR 0036](0036-expression-contract-for-inward-documents.md)'s contract.** It is an inward-facing operational document; its voice is governed by [`../operations/document-voice.md`](../operations/document-voice.md). Authoring sessions that add or modify rows audit the prose against [ADR 0036](0036-expression-contract-for-inward-documents.md), the same as any operational document.

- **Refinement under load is expected.** The first cross-pattern exercise of this contract — Phase 3's SQL/migrations row plus the build-readiness gate's first dogfooding pass — is the first real test. Friction surfaces around what counts as "Layer 1 source-of-truth" (one prose document or several? a CONTRACT.md per artifact?), unclear cases at the edge of patterns (is a one-line shell script a pattern or part of an existing one?), gate-stack false positives — these are refinement signals, ENGINE_LOG-tracked. The mechanism is calibrated to the gap between unguarded patterns and contracted ones; if a different pattern shape dominates, calibration changes.

- **No grandfathering for prior artifacts under newly-instantiated patterns.** When a pattern's row lands in the instantiation table, every artifact authored under that pattern from that point forward complies. Pre-existing artifacts may be retrofit (the way ADR 0038 retrofitted [`../tools/validate.py`](../tools/validate.py) at S-0026), or may remain pre-contract with explicit acknowledgement in the row's notes. The retrofit-or-acknowledge choice belongs to the session that lands the row.
