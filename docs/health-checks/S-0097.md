## Non-obvious finding(s)

> ≥1 required, per [`engine/operations/health-check.md`](../../engine/operations/health-check.md). Not on any mechanical scanner's output. The audit's own observation — what the AI/user noticed that no rule would catch.

### Non-obvious finding A — Soft-warn fire-without-acted-on is itself a posture-tracked-soft-warn candidate

The validator emits 31 soft-warn categories. Two of them (`mempalace_diary_read_skipped`, `mempalace_boot_query_skipped`) fire across multiple recent archives but get acted on at near-zero rate (1 and 2 commit-references each). This is mechanically distinct from `missing_rigor_score` (which fires 30/30 but IS acknowledged as Phase-5-carryover) and from `issue_collision` (which clears as Issues close).

What the scanner can see: per-session counts. What the scanner cannot see: the *meta-pattern* that some soft-warn categories quietly slip into "informational-only" status without ever being formally re-classed. [ADR 0042](../../engine/adr/0042-soft-warn-lifecycle-archive-canon.md)'s 3-of-5 persistent-warn surface is the structural mechanism for noticing — but persistent-warn surfaces alone are act-or-acknowledge prompts, not re-classification triggers. A category that fires 5/30 with 1 commit-reference is a category the project has implicitly demoted to noise without doing the explicit work of either retiring it OR documenting it as informational-only.

The forward-fit consequence: as routine batches under cadence-20 generate batch-converged telemetry, the rate at which categories drift into "informational-only" status will accelerate. Without an explicit re-classification path, the validator output becomes increasingly noisy and the audit's prose has to do correspondingly more triage work each cycle.

**Recommendation:** at audit closeout (or in a dedicated cleanup session), audit each of the 31 soft-warn categories against the audit-window's commit-reference rate; categorize each as (a) actively-tracked, (b) informational-only-accepted, or (c) retire-candidate. Document the categorization in [`engine/operations/tools-validate-interpretation.md`](../../engine/operations/tools-validate-interpretation.md) so the validator pipeline visibly distinguishes intent. Routes through User adjudication.

### Non-obvious finding B — The audit script's output template lags ADR 0057's contract

`engine/tools/health_check.py` was last touched substantively at S-0029 (per [ADR 0022](../../engine/adr/0022-periodic-project-health-checks.md) Consequences). [ADR 0057](../../engine/adr/0057-adversarial-stance-for-health-check-audits.md) at S-0085 added five contract elements (user-buffered execution, posture inversion, stats-as-proxy anti-pattern, ≥1 retire-candidate-recommendation, cold-context probe) and rewrote `docs/health-checks/TEMPLATE.md` to carry the new sections. The script was not re-aligned: it still emits `## Fit / ## Gaps / ## Dead weight / ## Bloat / ## MemPalace / ## Cadence calibration / ## Summary` — the old template shape.

This audit handles the gap by hand-authoring the report against the new template (replacing the script's auto-generated content entirely). That works because the script's role is "mechanical input" and the AI assembles the prose. But the contract drift between script output and TEMPLATE.md is invisible to the validator (the TEMPLATE.md is documentation, not a validated shape) and easy to miss for future cadence audits.

**Recommendation:** track this as either (a) script-update-to-emit-new-template (clean alignment; medium scope), (b) explicit documentation in [`engine/operations/health-check.md`](../../engine/operations/health-check.md) that the script's output is mechanical-input-only and the audit author assembles against TEMPLATE.md (zero scope; reaffirms current practice), or (c) deprecate the script's prose-emission entirely and emit JSON the AI consumes (largest scope; cleanest seam). Routes through User adjudication.

> Per [Issue #36](https://github.com/StarshipSuperjam/paideia/issues/36) and [ADR 0057](../../engine/adr/0057-adversarial-stance-for-health-check-audits.md). The audit reads `pushback`-tagged and `lesson`-tagged drawer content (not just count) since `last_audit_session`, surfaces clusters, recommends posture rules for mechanization when clusters appear.

### Pushback clusters

**Cluster A: Hedge-shaped close-summary language.** S-0071's [pushback] is the canonical instance: at session close, the AI authored a forward-looking note ("correctable in any future session via a JSON edit") that conveyed awareness without committing to a fix. The user pushed back: *"What session will pick up that JSON edit?"* The AI acknowledged the hedge was empty in exactly the way the standing memory `feedback_defer_indefinitely_is_pointless.md` flags. The pushback fired AFTER the memory was already written — posture-rule discipline is incomplete; the rule didn't catch the lapse in practice.

The pattern: at session close, when a post-close issue is discovered, the temptation is to write a forward-looking note that conveys awareness without committing. The note feels like discipline ("I noticed it") but is functionally identical to silence ("nobody will pick it up"). The right move when fix-now is blocked: file the gh issue *during* the session before the user has to ask.

S-0048's [pushback] is structurally adjacent — "preserved for manual review" framing on dirty-worktree handling. The user pushback: *"Manual review? This project has no manual processes. Everything is done here in session for auditability."* Same anti-pattern: deferral language that masks lack-of-resolution.

**Recommendation: validator-side posture-rule mechanization.** A new soft-warn `outcome_summary_hedge_keywords` could parse `outcome_summary` text on archive write for hedge-pattern keywords (`correctable in any future session`, `preserved for manual review`, `picked up by a future session`, `defer indefinitely`, `revisit when`, `manual cleanup`) and fire when matches occur without an accompanying Issue-filed reference (`#NNN` or `Issue #NNN`). The warn is informational at first fire; promote to hard-fail after [ADR 0042](../../engine/adr/0042-soft-warn-lifecycle-archive-canon.md)'s 10-session-persistence escalation criterion if the pattern persists.

Caveat: hedge-keyword scanning is brittle (false positives on legitimate forward-pointer prose). The safer formulation is a structured archive-field requirement: any `outcome_summary` claim of "future session will fix this" must carry a paired `next_session_handle: <issue-id-or-S-NNNN>` field that's grep-friendly for the validator. Routes through User adjudication.

**Cluster B: Single-point computation without bounded uncertainty.** S-0005's [pushback] is the only audit-window instance, but the lesson generalizes: when running parameter verification with under-specified inputs, default to a sensitivity sweep across the realistic range rather than a single-point computation. Single point = "trust me"; sensitivity sweep = "here's the regime where the answer changes."

The S-0005 instance was self-correcting (the AI ran the sweep after self-pushback, before the user surfaced concern). **No mechanization recommendation; the posture is exercised correctly when it's named.** Surface in [`engine/operations/code-discipline.md`](../../engine/operations/code-discipline.md) or migration-discipline.md if a third instance accumulates.

### Lesson clusters

The 3 tagged lesson drawers are sparse for the audit window (20 sessions). Plus the [lesson]-prefixed content in conversation-text matches mostly captures the diary-write-instructions template text appearing in transcripts (the pattern the `audit_mempalace()` script tries to detect via Tags-line match). **No actionable lesson clusters this audit;** the lesson convention is being exercised at low volume but appropriately (lesson capture is meant for non-obvious workflow gotchas, and the campaign sessions C1–C7 were highly procedural with few novel-discovery moments).

## Cadence calibration

Is the current cadence (20 sessions) right for current project velocity?

- If consistently no-action: raise (e.g., 10 → 20 → 30).
- If consistently large action lists: lower (e.g., 10 → 5).
- During phase transitions: consider one-off audit at the boundary.

**This audit's cadence-decision summary** (settled at session boot via user direction; recorded in `register_state.json:health_check_cadence_history` and [ADR 0022](../../engine/adr/0022-periodic-project-health-checks.md) S-0097 Consequences amendment): cadence raised from **10 → 20** effective S-0097.
