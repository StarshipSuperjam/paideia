## Non-obvious finding(s)

### Non-obvious finding A — HNSW recall degradation recurred *after* a full fix campaign; nothing catches the recurrence between audits

**Observation.** The S-0141 audit flagged MemPalace HNSW at "94% invisible to vector search." The project responded hard: [Issue #112](https://github.com/StarshipSuperjam/paideia/issues/112) (HNSW rebuild) closed at S-0144; [Issue #113](https://github.com/StarshipSuperjam/paideia/issues/113) (flush-primitive root cause) + ADR 0079 (sync-threshold tuning) + the mempalace 3.3.5 upgrade all landed at S-0145. Four S-0141 findings, all closed. Yet **this audit's freshness probe shows recall is degraded again**: `mempalace repair-status` reports HNSW `status: UNKNOWN` ("metadata has not been flushed yet"), and a direct query naming ADRs 0085–0088 — which landed at **S-0152, seven sessions after the rebuild** — retrieves none of their decision drawers. The fix addressed the *then-current* divergence; it did not make the index durably stay converged.

**Why it's non-obvious.** Every mechanical surface says the HNSW story is closed. Four Issues closed, an ADR landed, a version bump shipped. [Issue #109](https://github.com/StarshipSuperjam/paideia/issues/109) even wired `mempalace_hnsw_divergence` into in-session `validate.py` so the soft-warn flows into archive telemetry — and across the 20-session window that soft-warn is **absent from every archive**. But "metadata has not been flushed / UNKNOWN" is not a *divergence percentage* — the divergence check fires at ≥10% measured divergence, and an unflushed/UNKNOWN index produces no percentage to compare. The check that was added precisely to catch this class has a blind spot exactly where the current degradation lives. The recurrence is invisible to telemetry not because the telemetry failed to fire but because the degradation took a shape the threshold doesn't recognize.

**Recommendation.** Treat HNSW health as a **recurring** maintenance cost, not a one-time fix. Two parts: (a) extend the `mempalace_hnsw_divergence` probe (or add a sibling) to treat HNSW `status: UNKNOWN` / unflushed-metadata as a *suspect* signal in its own right, not only the ≥10% divergence path — so the recurrence surfaces in archive telemetry; (b) decide explicitly, before Phase 6 architecture work leans on recall, whether MemPalace semantic recall gets a durable fix or whether git-grep-against-tracked-files becomes the accepted Phase-6 recall substrate (the forward-fit map's open decision). Route (a) as a `health-check-finding` Issue; (b) is a user judgment call for STATE.md or a tension.

### Non-obvious finding B — `health_check.py`'s `audit_mempalace()` probe commits the stats-as-proxy anti-pattern it exists to catch

**Observation.** The `health_check.py --dry-run` mechanical emit for this audit reported: *"Decisions-room drawers (1) significantly below ADR count (88); ratio 0.01. Two-layer recording compliance gap."* A live `mempalace_list_rooms` call during this audit shows the `decisions` room holds **236** drawers — not 1. The script's probe is reading a wrong number (almost certainly querying a single wing while the upstream wing-naming bug, Issues #1/#2, has scattered drawers across 77 wings) and emitting a **false "two-layer recording compliance gap."** Two-layer recording is in fact *healthy* — 236 decision drawers against 88 ADRs, plus the post-adr-write hook reminder, plus archive `add_drawer_calls` telemetry showing per-ADR drawer writes (S-0152 wrote 4 for its 4 ADRs).

**Why it's non-obvious.** This is the [ADR 0057](../../engine/adr/0057-adversarial-stance-for-health-check-audits.md) stats-as-proxy anti-pattern *turned on the audit's own machinery*. ADR 0057 added freshness probes specifically because "counts and existence checks don't answer the operative diagnostic question." But `audit_mempalace()` is itself a count-emitter, and its count is wrong — and an audit that trusted the script's bullet (the lazy path the operative-discipline section explicitly warns against: "if you find yourself producing a categorized list quickly, you are reporting, not auditing") would chase a fabricated compliance gap, file an Issue to "audit which ADRs lack a decision companion," and burn a session on a non-problem. The probe built to feed the adversarial audit is itself plumbing emitting a misleading signal. Same shape applies to the script's `pushback`/`lesson` adoption-count probe, which returned "no output" this audit and also at S-0141 — two consecutive audits of a silent probe.

**Recommendation.** Fix or retire the `audit_mempalace()` room-count + tag-adoption-count probes. The room-count probe should either query across all wings (resilient to the #1/#2 wing scatter) or be removed — a probe that emits a false compliance gap is worse than no probe. The `pushback`/`lesson` CLI adoption-count path should be diagnosed or retired (it has produced no output for two audit windows). Route as a `health-check-finding` Issue. This is also the audit's **retire candidate** (see Affirmative retire candidates below).

### Non-obvious finding C — routine-mode procedure-doc drift is a *cluster*, and the recipe-vs-reference contract assumed only one drift direction

**Observation.** Five Issues in this window are the same class: [#122](https://github.com/StarshipSuperjam/paideia/issues/122) + [#123](https://github.com/StarshipSuperjam/paideia/issues/123) + [#124](https://github.com/StarshipSuperjam/paideia/issues/124) (closed S-0160) + [#125](https://github.com/StarshipSuperjam/paideia/issues/125) (discovered S-0157, open) + [#126](https://github.com/StarshipSuperjam/paideia/issues/126) (filed S-0161, open). Every one is routine-mode procedure machinery — a SKILL, an ops doc, or a command body — drifting out of sync with the canonical procedure or with a sibling. `start-routine.md` had become a stale 11-step subset of a 16-step procedure (#123); SKILL step 10 mis-described the commit shape (#122); the eager-claim step omits `task_id`/`target_id` (#126); `session-build-lifecycle/SKILL.md` is *missing* a boot step both its command and Layer-1 doc carry (#125). S-0160's own `lesson` drawer named the deeper pattern: *"'the Skill is the canonical form' is an aspiration, not a guarantee; drift can run command→skill too."*

**Why it's non-obvious.** Each Issue, taken alone, is a tidy one-line doc fix — and each got fixed that way. But [ADR 0044](../../engine/adr/0044-skill-conversion-recipe-vs-reference.md)'s recipe-vs-reference partition rests on a premise the window falsifies: that drift flows **doc → skill** only ("updates flow doc → skill, never the reverse"), so keeping the Layer-1 doc canonical and the skill a faithful mirror is sufficient. The window shows drift running **skill ↔ skill** (#125: build-lifecycle SKILL missing a step its siblings carry) and **command ↔ skill** (#123). No mechanical check verifies a SKILL is a faithful superset of its Layer-1 doc, or that sibling SKILLs agree on shared boot steps. The cluster is being fixed one Issue at a time while the generator of the cluster — un-checked multi-file procedure enumeration — keeps producing.

**Recommendation.** This is a pushback/lesson **cluster** (see Accumulated pushbacks and lessons) warranting a posture-rule-or-mechanization decision, not a sixth one-line fix. Options: a validator check that diffs each routine/build SKILL's enumerated boot-step set against its Layer-1 doc's step set and soft-warns on divergence; OR an ADR 0044 amendment that names the bidirectional drift vector and commits a same-session parity-check discipline. Route through User adjudication.
