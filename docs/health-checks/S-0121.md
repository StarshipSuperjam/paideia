## Non-obvious finding(s)

> ≥1 required, per [`engine/operations/health-check.md`](../../engine/operations/health-check.md). Not on any mechanical scanner's output. The audit's own observation — what the AI/user noticed that no rule would catch.

### Non-obvious finding A — Junk-drawer-prose accumulation is a structural AI-authorship default, not a per-file lapse

The user pre-loaded this observation at plan entry, but the structural shape is the audit's responsibility to name. The pattern: when a project has multiple capture-systems (ADRs, ENGINE_LOG, MemPalace `decision`/`pushback`/`lesson` drawers, session archives, git history), the AI's default authorship mode is **additive narrative inside the most-visible prose file**. Each session appends "what just happened" to STATE.md because STATE.md is read first; each ADR amendment appends inside the Consequences section because the ADR is the canonical decision home; each HANDOFF resolution appends a `**Resolved:**` block but doesn't prune.

The four-layer trace system per [ADR 0036](../../engine/adr/0036-expression-contract-for-inward-documents.md) is structurally designed to *prevent* this — but the discipline rests on AI posture. There is no validator soft-warn for "this governed document is accumulating session-narrative content." `audit_handoff_dispositions.py` audits new sections for disposition-line presence but doesn't audit for resolved-section pruning. `validate.py` doesn't audit ADR Consequences shape (only `adr_consequences_deliverable_audit` for promised deliverables).

The structural fix has three lanes (route through User adjudication):

1. **Validator soft-warn for STATE.md prior-row accumulation** — fires when STATE.md token count exceeds a threshold (e.g., 2000 tokens) OR when STATE.md gains a row with "(prior)" in its label. Easy to mechanize.
2. **Validator soft-warn for ADR Consequences inline-amendment pattern** — fires when an ADR file gains a `### Amendment` or `### Amendment (S-XXXX)` header inside its Consequences section. Requires structural parse of ADR body sections (modest).
3. **Validator soft-warn for HANDOFF.md long-resolved sections** — fires when a section carries a `**Resolved:**` line dated > N sessions ago AND remains in the file. Easy extension of `audit_handoff_dispositions.py`.

**Recommendation: land all three at the next interactive session that picks up audit findings.** Mechanizing the discipline prevents recurrence without depending on AI posture per session.

### Non-obvious finding B — The `ops-doc-uncited` axis in `scan_orphans.py` is itself stats-as-proxy-for-function

The dead-weight scanner flagged 19 of 22 operations docs as "not cited in last 20 session archives." Each flagged doc has 18–63 inbound *file* references; the scanner's signal is that no archive's `outcome_summary` literally contains the doc's path. But operations docs are *consumed by behavior* — sessions follow them — not by literal citation. The session that runs the shutdown sequence does not write "I read `session-shutdown-sequence.md` and followed it" into `outcome_summary`; the session *follows the sequence*.

The scanner's `ops-doc-uncited` axis is therefore **mostly false-positive for foundational ops docs**. It correctly flags ops docs whose role *is* explicit citation (e.g., a sub-procedure ops doc whose only consumer is another doc) but mis-flags load-bearing core ops docs.

**Recommendation: refine the axis.** Two options (route through User adjudication):

1. **Drop the axis entirely** for ops docs with > 10 inbound *file* references — the references themselves are the consumption signal. Keep the axis for docs with < 10 inbound references.
2. **Replace the axis** with a richer signal: ops docs not cited in any session archive AND not transitively reachable from `engine/STATE.md` OR `CLAUDE.md`. A doc with no inbound references from STATE/CLAUDE is genuinely orphan; a doc reachable from the boot-ceremony surface is consumed by behavior.

The scanner's *other* axes (`reference-count`, `last-mod-age`, `register-empty`, `stale-marker`) appear to produce useful signal — they're not flagged here. Only `ops-doc-uncited` shows the failure mode.

### Non-obvious finding C — Routine-mode batch sessions produce near-zero `pushback` / `lesson` capture

The MemPalace probe surfaced 45 `pushback`-text-match drawers and 45 `lesson`-text-match drawers across project history. Inspection: the bulk of `pushback` drawers cluster around interactive-session push-gate gotchas (S-0003 / S-0059 / S-0060) and the standing-pushback-rule recapture pattern; the bulk of `lesson` drawers are *meta* — they recite the shutdown-instruction prose for filing pushback/lesson drawers, not actual lessons. Genuine pushback / lesson content lands in **interactive** sessions; the routine block (S-0104 → S-0120, 14 of 23 audit-window sessions) appears to produce near-zero pushback/lesson capture.

This is a tension with [ADR 0057](../../engine/adr/0057-adversarial-stance-for-health-check-audits.md) — the audit's pushback/lesson cluster surface depends on the captures being made. If routine-mode produces no captures, the audit-window's effective surface shrinks to interactive sessions only. Two diagnostic hypotheses (not mutually exclusive):

1. **Routine-mode posture suppresses pushback / lesson capture by design.** No user-interaction surface means no in-thread pushback exchange to capture; the constrained scope means no procedural-failure-mode discovery to record as a lesson. The drawer types are interactive-session-shaped.
2. **The shutdown procedure mechanizes pushback/lesson filing as a discipline, but routine sessions execute the procedure mechanically without the substantive content.** The drawer gets filed but its content is meta-instruction recitation rather than real capture.

**Recommendation: the audit doesn't adjudicate which is which without more evidence.** Route through User adjudication; either disposition has a concrete fix (hypothesis 1: document the expectation that routine sessions skip pushback/lesson capture, no fix needed; hypothesis 2: refine the shutdown procedure's pushback/lesson capture prompt to ask "did anything *substantive* surface this session?" with explicit permission to skip if no).
