## Non-obvious finding(s)

### Non-obvious finding A — MemPalace retrieval is broken in three reinforcing ways, not one

**Observation.** S-0162 named "MemPalace HNSW recall degradation recurred" and routed the telemetry-extension fix to [Issue #127](https://github.com/StarshipSuperjam/paideia/issues/127) (closed S-0163). This audit's freshness probe finds three reinforcing breakage modes co-existing:

1. **HNSW UNKNOWN.** `mempalace repair-status` reports `status: UNKNOWN` ("metadata has not been flushed yet") — same shape S-0162 found. The S-0163 #127 telemetry catches it (`mempalace_hnsw_status_suspect` fires), but doesn't fix it. Vector search runs against a degraded surface.
2. **Wing/room-filtered search throws upstream error.** `mempalace search --wing paideia --room pushback` returns `Internal error: Error finding id` — upstream [Issue #1](https://github.com/StarshipSuperjam/paideia/issues/1) (open since 2026-05-04). The audit's prescribed `pushback`/`lesson` tag-scoped query workflow per [ADR 0057](../../engine/adr/0057-adversarial-stance-for-health-check-audits.md) element 4 is structurally un-runnable.
3. **BM25 fallback returns mining pollution.** With vector search degraded, BM25 dominates; with the wing scatter (98 wings now, up from 77 at S-0162) carrying captured terminal output, BM25 matches against captured `[soft-warn:...]` strings and pre-commit hook output rather than curated decision/lesson drawers. **The audit's three representative queries returned validator soft-warn dumps as top hits.**

**Why it's non-obvious.** S-0162 isolated #1 and added telemetry; #2 was a known upstream open issue; #3 was named in S-0162 as a separate concern (`mempalace_wing_count_growth` annotated baseline). Each individually looks like a contained, prioritized item. Together they compound: the user-named question "do we need a recurring fix routine?" gets a sharper answer — **a recurring fix routine that rebuilds HNSW would not address modes 2 or 3**, because flushing the index doesn't unblock wing-filter and doesn't un-pollute BM25 with mining content. Recurring maintenance pays cost on infrastructure that has three independent breakage paths, only one of which the maintenance addresses. See user adjudication of Finding-MEMPALACE for the structural alternative.

**Recommendation.** Route Finding-MEMPALACE to the user adjudication below; the audit's recommendation leans toward **Finding-MEMPALACE option (c)** (settle the Phase-6 recall-substrate decision now) because options (a/b/d) all pay maintenance cost on a stack with three independent breakage modes.

### Non-obvious finding B — the persistent-warn surface is saturated; the alarm-too-noisy-to-alert pattern

**Observation.** Seven of the validator's soft-warn categories are at the persistent-warn 10-of-N escalation threshold per [`soft-warn-lifecycle.md`](../../engine/operations/soft-warn-lifecycle.md): `mempalace_wing_count_growth` (30/30), `issue_collision` (30/30), `missing_rigor_score` (30/30), `edge_evidence_empty` (30/30), `mempalace_zero_citations_after_search` (25/30), `skill_layer1_parity_drift` (21/30), `state_md_row_count` (15/30). Per the cadence-fired audit posture each category requires *promote-to-hard-fail*, *accept-and-annotate*, or *address-inline* dispositions. **All seven currently sit at "accept-and-annotate"** — they ride along in the persistent-warn surface at every boot, the user has acknowledged them, no one acts on the surface.

**Why it's non-obvious.** Each category has a defensible reason for being annotated baseline (`mempalace_wing_count_growth` waits on upstream fix; `issue_collision` was already refined at S-0143; `missing_rigor_score` + `edge_evidence_empty` are Phase-5-residue + Phase-6-rework targets; `mempalace_zero_citations_after_search` was folded into Issue #127; `skill_layer1_parity_drift` waits on Issue #125 + the `session-build-lifecycle` SKILL parity gap; `state_md_row_count` was inline-pruned at S-0162 and grew back). Reading the persistent-warn surface entry-by-entry, every annotation has a story and the story is plausible. **But the aggregate effect is that the persistent-warn surface that was authored to *alert* the project is now too noisy to alert anything.** The S-0163 `skill_layer1_parity_drift` mechanism is the canonical example — the validator added a check, the check fires every commit, the gap it names (Issue #125) is open, and no session has acted on it because the surface is full of similar-shaped "open issue, persistent fire" annotations. The mechanism that should have closed Issue #125 within a couple of sessions instead just adds another line to a saturated surface.

**Recommendation.** **The persistent-warn surface needs a contract revision, not category-by-category triage.** Two structural options to surface to the user:
- **(B1) Annotation-as-disposition makes the surface inert** — convert annotated-baseline categories to a separate "annotated-baselines" bucket that doesn't fire the boot persistent-warn surface. Truly-inactionable persistent-warns get formally documented as inactionable (in `tools-validate-interpretation.md`'s "Persistent-warn annotation" section) and stop competing with action-needed ones. The boot surface returns to its alerting role.
- **(B2) Categories without an active resolution path get retired or converted to hard-fail** — for each of the seven, the audit asks: is there an active resolution (Issue, in-flight work)? If yes, the category stays. If no (truly inactionable for the project's lifetime), retire it. If the resolution should fire faster than annotation accommodates, promote to hard-fail.

Both options are session-sized. Route as a `health-check-finding` Issue with the option choice for the next interactive session. See user adjudication.
