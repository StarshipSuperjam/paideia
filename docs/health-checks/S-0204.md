### Non-obvious finding A — ADR 0091 retirement-residue cluster (session-start hook still probes the retired MemPalace MCP at every boot)

**Observation.** The session-start.sh hook ([`engine/tools/hooks/session-start.sh`](../../engine/tools/hooks/session-start.sh)) carried three full sections probing infrastructure retired by ADR 0091:

1. **MemPalace MCP availability probe** (per S-0089 follow-up to S-0087/S-0088). 108 lines. Probed `command -v mempalace` + `mempalace status`. Post-ADR-0091 the mempalace CLI is gone; the probe fired `mempalace-mcp=cli-missing` at every boot (visible in this session's boot log: `2026-05-18T14:42:19Z FAIL reason=mempalace-mcp=cli-missing`). The hook downgraded the FAIL to soft `mcp=cli-missing threshold=skipped-prereq-or-mcp`.

2. **MemPalace HNSW sync_threshold consistency probe** (per ADR 0079, S-0145). 88 lines. Gated on `MCP_PROBE_STATUS=ok` which was never `ok` post-ADR-0091. Dormant but live.

3. **Pending mempalace diary writes** (per ADR 0056, S-0091). 50 lines. Read `engine/session/diary_pending_index.json` (which existed with `pending: []`) and referenced the retired `mempalace_diary_write` recovery flow.

Additional residue surfaces:
- [`engine/tools/probe_versions.py`](../../engine/tools/probe_versions.py) probed `chromadb` + `mempalace` versions at every boot, both reporting `not-installed` post-ADR-0091.
- [`engine/tools/probe_palace.py`](../../engine/tools/probe_palace.py) referenced in docs + comments — file already deleted, the references dangle.
- [`.claude/logs/mempalace-hook.log`](../../.claude/logs/mempalace-hook.log) + `post-mempalace-tool-use.log` stale since 2026-05-13/14, accumulating without ever being touched again.

**Why it's non-obvious.** Each individual site looks like contained legacy comment-drift. Together they form a cluster — the substrate-flip "demolition pass" at S-0193 per ADR 0091's T1-E criterion was structurally complete for `engine/tools/mempalace_*.py` deletions + validate.py check renaming, but missed the session-start hook surfaces entirely. The boot surface appeared healthy because the hook absorbed the FAIL as soft warning; no downstream signal escalated. **The S-0193 deliverable's "no live mempalace tooling, no live mempalace hooks" was true for tools but false for hooks** — the cleanup pass searched `engine/tools/mempalace_*.py` patterns and missed the broader hook surfaces.

This is also the **stats-as-proxy-for-function anti-pattern** ([ADR 0057](../../engine/adr/0057-adversarial-stance-for-health-check-audits.md)): the boot probe reports `mcp=cli-missing` which is mechanically a soft signal, not a failure. The number doesn't escalate. But the operative diagnostic — *is this probe doing the work it was created to do?* — answers no. The probe was protecting against an escape-hatch token mis-use that no longer matters because the token + substrate both retired.

**Recommendation.** Inline cleanup this session — strip the 3 retired sections from session-start.sh (243 lines), update stale comments throughout, retire `chromadb` + `mempalace` version probes in probe_versions.py + tests, delete `diary_pending_index.json` + 4 stale log files. Routes through User adjudication — accepted, executed inline. See commit `435c789`.

### Non-obvious finding B — validate-history.jsonl per-phase schema regression (ADR 0063 runtime-regression check structurally inoperative)

**Observation.** [`validator_runtime_phase_regression`](../../engine/operations/tools-validate-interpretation.md) per [ADR 0063](../../engine/adr/0063-validator-tiered-runtime-targets-and-regression-soft-warn.md) fired **0 times across the entire S-0184 → S-0203 audit window** (20 sessions). The tiered targets — structural <500ms, health_probe <5s, graph_audit <5s, total <11s — are unmonitored.

Root cause traced:

1. **Boot-probe path explicitly skips per-phase fields** (`validate.py:4646-4654`). The conditional block writes `duration_structural_ms` / `duration_health_probe_ms` / `duration_graph_audit_ms` only in the default-mode pipeline — `--health-probe-only`, `--code-gates`, `--sql-gates`, `--export-snapshot` paths intentionally skip these fields with the comment "Per-phase fields are only meaningful for the default-mode pipeline."

2. **The canonical parent-repo `engine/tools/validate-history.jsonl` only accumulates boot-probe entries.** Across 90 recorded entries, the schema is `["checks_run","duration_total_ms","hard_fails","session_id","soft_warns","timestamp"]` — boot-probe shape, no per-phase fields. Latest 3 entries are from boot-probe writes (S-0186 / outside-session / S-0190). Default-mode pre-commit hook validate.py runs (33 checks per run, ran 2× this session at eager-claim + Finding A commit) do not appear to land in the canonical file.

The regression check at `validate.py:4275-4378` requires the per-phase fields to compute; absent them, it skips silently.

**Why it's non-obvious.** The check function runs (visible in `checks_run` arrays); the soft-warn category exists; the validate-history.jsonl has 90 entries; everything looks operational. The stats-as-proxy-for-function gap is at the data-shape layer — the records being written aren't the records the check needs.

**Recommendation.** File as `health-check-finding` Issue with three investigation options (extend boot-probe schema; re-route default-mode entries to the canonical file; retire the check). Investigation required before fix design. Routed through User adjudication — accepted, [Issue #150](https://github.com/StarshipSuperjam/paideia/issues/150) filed in-session.
