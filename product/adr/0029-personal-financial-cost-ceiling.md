# ADR 0029 — Personal financial cost ceiling is an operating constraint

- **Status:** Accepted
- **Date:** 2026-04-29
- **Deciders:** S-0008 (exploration carried into the build session — operating-discipline commitment surfaced during BYOK assessment)

## Context

Paideia is a private project with zero outside funding. The builder's situation rules out two of the three pressure-release valves a typical SaaS would use to absorb unbounded marginal cost: there is no investor capital to subsidize early users, and there is no institutional backstop (the project is not yet a non-profit per [`docs/business.md`](../docs/business.md) Non-Profit Incorporation, and grant funding requires 501(c)(3) status that itself is deferred until grants are actually in scope). The third valve — "education apps with AI-driven marginal costs are not the kind of company VCs fund" — closes off the future where this changes by raising money. The funding situation is structural, not transitional.

The cost shape is not benign. [`docs/business.md`](../docs/business.md) (Cost Model) estimates $100–120k/year operating cost at 10,000 users, dominated by API spend. A single power user working through Hegel can burn significant per-month API cost; at $15–20/month consumer pricing, the system loses money on the exact users it most wants to serve. The Seasonal Cost Profile section adds that real costs concentrate in three ~12-week academic bursts per year — meaning peak-quarter costs run substantially higher than the annualized average. The Personal-Use Cost Estimates section names the floor: ~$3–7/month for the builder's own use with prompt caching, manageable on a personal budget. Everything between "personal floor" and "ten-thousand-user steady state" is a gradient where the cost crosses the line from "I pay my coffee bill" to "I am personally on the hook for several thousand dollars I did not consent to."

The crossing is asymmetric. There is no value of an unmonitored 10x usage spike that justifies the personal financial exposure. The expected value of "open access, eat the cost" is negative for the builder regardless of how positive the spike's *outcome* might be: a viral moment that delivers $4,000 of value to learners while delivering a $4,000 surprise bill to the builder is not a success — it is a failure mode. The asymmetry is the load-bearing observation: cost protection is not a "nice to have when convenient"; it is a *precondition* for opening the system to non-builder use at all.

The forcing function is launch sequencing. Phase 8 is when external evaluation begins; Phase 9 is when the UI prototype reaches anything resembling a usable surface. Both phases involve at least small numbers of non-builder users. Without the cost-ceiling commitment recorded as a *gating* constraint before those phases begin, the natural drift is to ship a prototype with no cost-cap mechanism and discover the gap empirically — exactly the failure mode this ADR exists to forbid. The commitment must be in place before any code path that calls the Anthropic API on behalf of a non-builder user lands.

## Decision

**The system must not be exposed to any non-builder user without a hard cost-ceiling mechanism in place.** The mechanism's job is to bound the builder's personal financial exposure to a value the builder has explicitly accepted in advance — never to discover the cost ex-post.

Three architectural commitments follow from this principle.

**(1) Cost protection is a precondition, not a feature.** No phase that opens the system to non-builder users may close without a cost-cap mechanism wired and tested. This applies to Phase 8 (evaluation harness, even if the evaluators are a small instructor sample), Phase 9 (UI prototype with any external testers), and any subsequent expansion. "Cost-cap mechanism" means at minimum: a per-user spend ceiling, an aggregate-system spend ceiling, telemetry that reports current spend in real time, and a defined behavior at the cap (per the soft-wall question deferred to OQ-WALL-BEHAVIOR — see [`docs/tensions.md`](../docs/tensions.md)). The specific values, the telemetry shape, and the cap-behavior are deferred; the requirement that *some* mechanism exists is not.

**(2) Architectural choices are evaluated against this constraint.** Features that create uncapped marginal cost are constrained or rejected. The institutional BYOK regime under consideration in OQ-BYOK-REGIME (see [`docs/tensions.md`](../docs/tensions.md)) is *attractive specifically because* it shifts the cost surface from the builder to the institution — the institution holds the API key, the institution holds the cost. The seasonal pre-buy revenue pattern under consideration in [`docs/business.md`](../docs/business.md) (the tier-and-pack reframing) is attractive specifically because it pre-collects revenue against the quarterly cost spikes the Seasonal Cost Profile names. Features that would expand the cost surface without expanding the protection — for example, a "ask Paideia anything" general chat surface that bypasses the bounded teaching contexts ([ADR 0028](0028-input-side-scope-structural-not-prompt.md)) — are foreclosed by this ADR's interaction with [ADR 0028](0028-input-side-scope-structural-not-prompt.md), not just by [ADR 0028](0028-input-side-scope-structural-not-prompt.md) alone.

**(3) The cap-behavior must preserve learner experience within the limits the cap allows.** The "soft walls degrade, don't terminate" principle: when a user approaches their per-user cap, the agent should downshift gracefully (Opus → Sonnet, deeper retrieval window → shallower, two-hop neighborhood → one-hop) rather than terminating mid-concept-engagement. The atomic unit of teaching (per [`docs/session-lifecycle.md`](../docs/session-lifecycle.md), Concept Engagement as the Atomic Unit) is the concept engagement, which spans hours or days across multiple sittings. A wall that fires mid-engagement violates that integrity. The specific degradation behavior is deferred to OQ-WALL-BEHAVIOR; the principle that termination-mid-engagement is the wrong default is committed here.

This commitment does not specify the cost ceiling's numeric value. The value is a personal threshold the builder sets and may revise; recording it in an ADR would commit a number to a public artifact that does not need to be public. The value is recorded operationally (in the Anthropic billing console, in the cost-cap mechanism's configuration) rather than in the ADR.

## Consequences

- **Phase 8 and Phase 9 success criteria expand.** Phase 8 (evaluation harness) cannot close without a cost-cap mechanism wired and tested against the evaluation users' usage shape. Phase 9 (UI prototype) cannot open to external testers without the same. [`ROADMAP.md`](../../ROADMAP.md) Phase 8 and Phase 9 success criteria are extended in this session to reflect this gating.

- **The institutional path becomes preferred for non-builder access.** The institutional BYOK regime ([`docs/business.md`](../docs/business.md), Audience vs. Market section + the BYOK assessment captured in this session) is the simplest architecture consistent with this ADR — institution holds the API key, cost flows institution → Anthropic, the builder's personal exposure is structurally bounded to the builder's own use. Consumer launch (general public, BYOK or not) requires either a cost-cap mechanism robust enough to handle the consumer-tier failure modes or a deliberate decision to absorb the residual risk, recorded in a superseding ADR.

- **Soft-wall behavior is committed in principle; specifics open.** OQ-WALL-BEHAVIOR (added in this session to [`docs/tensions.md`](../docs/tensions.md)) settles the specific mechanism. The principle (degrade, don't terminate) is not deferred — it is committed by this ADR. The specific degradation ladder (Opus → Sonnet → smaller retrieval window → … → soft refusal with explanation) is the open question.

- **The "ask Paideia anything" general chat surface is doubly foreclosed.** [ADR 0028](0028-input-side-scope-structural-not-prompt.md) forecloses it on input-side-scope grounds (no general chat input). This ADR forecloses it on cost grounds (general chat is the canonical uncapped-marginal-cost surface). Both ADRs would need to be superseded to reopen it — and the superseding decision would need to specify how the cost-cap mechanism handles the new surface.

- **Internal fine-tuning as cost mitigation gains a load-bearing rationale.** [`docs/business.md`](../docs/business.md) (Internal fine-tuning from session data) names this as a candidate cost mitigation. Constrained by [ADR 0026](0026-persistent-learner-storage-structural-not-substantive.md) (no transcript-based fine-tuning), the path forward is fine-tuning on the structured event log and structured tension records — which can produce a cheaper Sonnet-class model handling routine teaching, with Opus reserved for cross-domain edge generation and self-correction. This ADR does not commit to fine-tuning as the path; it raises the priority of evaluating it because cost protection is a hard constraint, not a target.

- **This commitment does not rise to [`docs/MISSION.md`](../docs/MISSION.md)'s strong-working-commitments list.** Consistent with the same reasoning [ADR 0026](0026-persistent-learner-storage-structural-not-substantive.md) used: that list is the audience-facing summary of pedagogical commitments — what makes Paideia different as a teaching tool. The cost ceiling is operating-architecture discipline, not a teaching commitment. Cross-references from [`docs/business.md`](../docs/business.md) and [`ROADMAP.md`](../../ROADMAP.md) Phase 8 / Phase 9 are sufficient. Commitment 2 in [`docs/MISSION.md`](../docs/MISSION.md) ("Commercial sustainability without pedagogical compromise") covers the audience-facing version of the principle; this ADR specifies the operating constraint that commitment 2 rests on.

- **The ceiling is a fixed annual operating subsidy budget — nothing ever raises it.** *Reframed in S-0012 by [ADR 0032](0032-personal-project-disposition.md).* The original framing of this consequence was bridge-to-grants: until a grant landed, the cost-ceiling mechanism protected the builder; after a grant landed, the ceiling could be raised. ADR 0032 retracts that framing as part of foreclosing the institutional / grant / acquisition option set. The ceiling is now a fixed annual operating subsidy budget the builder commits in advance and does not exceed. If usage growth would push spend past the budget, the response is to raise prices to slow growth, cap signups, or pause the App Store listing — never to pursue institutional clients, grants, or acquisition (per ADR 0032's binding sentence). The mechanism (per-user cap, aggregate cap, real-time telemetry, soft-walls-degrade-don't-terminate) is unchanged. The numeric ceiling value remains private operational configuration; only the value's *justification structure* changed (subsidy budget, not bridge-to-grants).

- **Backup tag and rollback discipline gain a new use case.** The `pre-foundation-v0.0.0` backup tag (per [`STATE.md`](../../engine/STATE.md) Backup tag) is a "rollback the project" mechanism. This ADR adds a related operational mechanism: a "rollback non-builder access" capability — the ability to revoke API key exposure rapidly if the cost-cap mechanism fails. Specification belongs to Phase 8 deployment work, not this ADR.

- **The cost-ceiling value remains private.** The numeric ceiling and the cost-cap mechanism's parameters are operational configuration, not public design artifacts. The ADR records the principle; the configuration records the numbers.

- **Operating in this regime is a deliberate trade.** The system will reject some users at the soft wall who would have been profitable customers at scale. The system will degrade quality for some users who would have benefited from the higher tier. Both are acceptable costs of bounded personal exposure. The alternative — open access with no cap — has expected value that is dominated by the personal-financial-risk failure mode regardless of upside. This trade is the load-bearing decision; the mechanisms that implement it are downstream.
