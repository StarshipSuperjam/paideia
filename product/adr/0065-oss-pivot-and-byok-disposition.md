# ADR 0065 — OSS pivot and BYOK disposition; open-source under Apache 2.0, users bring their own Anthropic key

- **Status:** Accepted
- **Supersedes:** [ADR 0029](0029-personal-financial-cost-ceiling.md), [ADR 0035](0035-multi-platform-apple-expansion.md)
- **Date:** 2026-05-11
- **Deciders:** S-0128 (formal recording of the OSS-pivot-plus-BYOK direction shift; first session of the three-session refactor — Session A lands the contract supersession; Session B rewrites downstream docs; Session C lands the LICENSE / PII cleanup / OSS hygiene / public-flip cycle)

## Context

The personal-project disposition committed at S-0012 by [ADR 0032](0032-personal-project-disposition.md) — and the platform-broadening that preserved it at S-0015 in [ADR 0035](0035-multi-platform-apple-expansion.md) — rests on two operating assumptions that no longer hold. **(i)** Paideia ships as a private codebase the builder distributes only through the Apple App Store. **(ii)** The builder eats marginal API costs within a fixed annual operating subsidy budget, recovered through a cost-priced subscription. Both assumptions held while the project was a viable solo-commercial venture. As of S-0128 the builder's read is that Paideia is not commercially viable in the form ADR 0032 / ADR 0035 imagined, and the right disposition is to ensure the artifact survives the builder's eventual attention loss rather than to optimize a revenue model that does not pay back the work it requires.

Two consequences follow simultaneously and cannot be separated.

**Open-sourcing the project is the artifact-survival move.** If the builder stops maintaining Paideia, a private codebase dies with the maintainer's interest. A public codebase under a permissive license can be forked, extended, and operated by anyone who finds the pedagogical contract worth carrying. The artifact has standalone value — the curated graph, the rendering-policy contract per [ADR 0027](0027-rendering-policy-prompt-layer-contract.md), the engine session-build apparatus, the dependency-graph pedagogy — independent of whether the builder personally keeps shipping it. Making the source open does not foreclose the App Store release; it strictly broadens the artifact's future surface area. The license choice is Apache 2.0 (permissive + explicit patent grant + NOTICE-file attribution survival; compatible with App Store distribution where AGPL has known tivoization-tension friction).

**BYOK is the only architecturally honest shape under public-source distribution.** Once the source is public, the builder's personal Anthropic API key cannot ship embedded in the App Store binary — it would be extractable from any installed app, and would still be the builder's billing exposure even if it weren't. The only choices are (a) BYOK with the user supplying their own Anthropic API key, or (b) abandoning the App Store distribution entirely. (a) is the survivable path. This inverts [ADR 0035](0035-multi-platform-apple-expansion.md) commitment 4 ("No bring-your-own-key, neither consumer nor institutional"), which itself inherited from [ADR 0032](0032-personal-project-disposition.md). That foreclosure was load-bearing under the original disposition: consumer BYOK was excluded specifically because it self-selects technically sophisticated users away from the freshman audience per [ADR 0012](0012-freshman-defaults-autodidact-ceiling.md). That argument is mitigable, not absolute — an in-app onboarding flow that walks a non-technical user through "create an Anthropic account → console.anthropic.com → API Keys → create → paste here" turns the technical-self-selection concern from a structural failure mode into a UX problem with known solutions. Multiple App-Store apps in this category have shipped successfully with non-technical user bases (Anthropic API-key clients including various BYOK AI assistants, plus precedents in adjacent categories like SSH clients and self-hosted productivity apps). The freshman-defaults calibration per [ADR 0012](0012-freshman-defaults-autodidact-ceiling.md) survives intact — it now applies to the in-app key-onboarding flow as well as to the teaching surface.

**[ADR 0029](0029-personal-financial-cost-ceiling.md)'s cost-ceiling mechanism is obsolete under BYOK, not merely reframed.** This is the critical distinction from the S-0012 amendment to ADR 0029 (when ADR 0032 retracted the bridge-to-grants framing). At S-0012 the *mechanism* (per-user spend ceiling, aggregate-system spend ceiling, real-time spend telemetry, defined cap-behavior, soft-walls-degrade-don't-terminate) survived intact — only the *justification structure* of the ceiling's value changed (subsidy budget rather than transitional protection). The S-0012 decision drawer is explicit that this called for inline amendment of ADR 0029, not supersession, because the mechanism remained the same. **Under BYOK the mechanism itself disappears.** There is no Paideia-controlled per-user spend ceiling because Paideia does not hold the user's API key. There is no aggregate-system spend ceiling because there is no aggregated Paideia API account. There is no Paideia-side real-time spend telemetry because the API calls do not transit Paideia infrastructure. The user's own Anthropic billing console is the spend-cap surface, but it is not a Paideia mechanism — it is the third-party billing relationship between the user and Anthropic. The pedagogical principle the cost ceiling protected (degrade rather than terminate mid-concept-engagement) survives as a teaching discipline; the cost-cap *apparatus* does not. This is the supersession trigger: mechanism changes, not just justification.

**The corruption-vector test ADR 0032 recorded still binds.** "Leave the option open to expand later" was the seed pattern that pulled design toward business outcomes pedagogy never warranted. The test: *does this proposal preserve optionality whose value depends on a pedagogical or operational direction the builder no longer commits to?* The OSS+BYOK shift passes the test. It does **not** reopen institutional licensing, grant funding, non-profit incorporation, or acquisition as exit — those remain foreclosed by refusal, not deferral. It removes the builder-funnel-mechanic threat the original ADR 0032 commitments were guarding against — because the builder no longer recovers revenue from API spend, there is no per-user funnel to optimize. The threat shape inverts: under OSS+BYOK the failure mode is **user-funnel-mechanic** (engineering retention nudges that bait users into more API spend on their own dime), and the same discipline answers it. Static polish, no streaks, no leaderboards, no push notifications beyond auth/billing, no community surface, no in-app messaging — commitment 6 of the prior disposition transfers verbatim and the new ADR preserves it.

The forcing function is timing. The repo cannot flip to public while it contains the builder's personal absolute paths and the LICENSE asserts "all rights reserved." The App Store binary cannot ship while it would expose the builder's personal API key. Both blockers are removed by the same disposition shift. This session lands the contract; downstream sessions land the doc rewrites and the cleanup. No code change is in scope here.

## Decision

**Paideia is an open-source project distributed under Apache 2.0. The official Apple App Store binary is maintained by the original maintainer under their personal Apple Developer account. Users supply their own Anthropic API key; the user's key never transits any Paideia-controlled infrastructure. The pedagogical commitments, the platform commitment, and the operating-discipline guard against funnel mechanics survive from [ADR 0035](0035-multi-platform-apple-expansion.md); the cost-ceiling mechanism per [ADR 0029](0029-personal-financial-cost-ceiling.md) retires; commitments 2 and 4 from [ADR 0035](0035-multi-platform-apple-expansion.md) invert; commitments 7 (OSS) and 8 (key-handling architecture) are new.**

The disposition has eight binding commitments. Commitments 1 and 6 preserve from [ADR 0035](0035-multi-platform-apple-expansion.md) verbatim. Commitments 2, 4, and 5 revise or retire. Commitment 3 retires and is replaced. Commitments 7 and 8 are new.

**(1) Apple platforms via App Store. iPhone + iPad first-class via a single SwiftUI codebase. Mac via Designed-for-iPad opt-in with modest keyboard/menu polish.** *Preserved verbatim from [ADR 0035](0035-multi-platform-apple-expansion.md) commitment 1.* Distribution is the Apple App Store. iPhone and iPad targets share one SwiftUI codebase. Mac distribution is via the "Designed for iPad" opt-in. Apple Developer Program enrollment ($99/yr) covers all three targets. No Android, no web app, no native AppKit/Catalyst Mac app. Reopening any of these foreclosed targets requires superseding this ADR.

**(2) Free download with a one-time In-App Purchase unlock. No subscription. Specific dollar amount deferred to Phase 8/9 implementation session.** *Inverts [ADR 0035](0035-multi-platform-apple-expansion.md) commitment 2.* The subscription model the prior commitment specified was structurally tied to API-cost recovery — Paideia's revenue had to scale with per-user marginal API expense. Under BYOK there is no marginal cost to recover; the user pays Anthropic directly. The app artifact (curated graph, teaching policy, syllabus generator, UX) is a one-shot deliverable with periodic updates, structurally aligned with a one-time-purchase model rather than recurring billing. The dollar amount is left TBD because the right price point depends on the in-app key-onboarding UX burden — which is unknown until the onboarding flow is built and exercised — and on whether the App Store cohort responds to "$x.xx for an app that requires an external service subscription" pricing as it would to a pure self-contained app. A free-tier of the app (graph browse without key, sample syllabus from static fixture) is a permitted structural affordance under BYOK because it costs the builder near-zero (no API spend; only the marginal infrastructure cost of the Supabase reads, which is dwarfed by the Apple Developer Program floor). **The free-tier is permitted as a structural affordance — it must not be engineered as a conversion funnel.** No streaks, no "you'd unlock more if you upgraded" nudges, no retention-pressure surfaces, no engagement-time-on-free-tier metrics treated as optimization targets. The discipline against funnel mechanics that commitment 2 of the prior disposition imposed survives unchanged; only the structural shape (free-with-IAP-unlock instead of paid-subscription) changes.

**(3) Builder's operating subsidy is the fixed infrastructure floor (~$200–300/year), independent of user count. Users pay Anthropic directly for their own API usage.** *Retires and replaces [ADR 0035](0035-multi-platform-apple-expansion.md) commitment 3.* The original commitment had the builder eating API costs against a fixed annual operating subsidy budget that *was* expected to cover marginal API for paid users at the design margin. Under BYOK the builder is no longer in the API-cost path. The remaining builder costs are the Apple Developer Program ($99/yr), Supabase (free tier for mastery state at expected scale; minimal paid tier if growth requires), and the domain registration for the Paideia.app or equivalent canonical web pointer. Total infrastructure floor lands ~$200–300/year and does not scale with user count — Supabase queries for mastery state read/write are negligible per user, and the Apple Developer Program floor is independent of how many users install. Users pay Anthropic directly through their own account. **There is no Paideia revenue path that recovers API cost.** Should the App Store IAP unlock per commitment 2 generate revenue, the revenue is for the app artifact (the developer's work building and maintaining the app), not for API-cost recovery — because there is no API cost for Paideia to recover.

**(4) Consumer BYOK is the required regime. Institutional licensing, grants, and acquisition remain foreclosed.** *Inverts [ADR 0035](0035-multi-platform-apple-expansion.md) commitment 4 on consumer BYOK; preserves the institutional-foreclosure half of that commitment unchanged.* Users supply their own Anthropic API key via an in-app onboarding flow. The technical-self-selection concern that excluded consumer BYOK under [ADR 0032](0032-personal-project-disposition.md) (consumer BYOK self-selects technically sophisticated users away from the freshman audience per [ADR 0012](0012-freshman-defaults-autodidact-ceiling.md)) is now treated as a UX problem with a known solution shape: the in-app onboarding flow walks a non-technical user step by step through Anthropic account creation, console navigation, key generation, and key entry, with screenshots, plain-language copy honoring the freshman-defaults calibration, and a "test the key" round-trip against `/v1/messages` before the user proceeds. **Institutional BYOK, institutional licensing, grant funding, non-profit incorporation, and acquisition as exit remain foreclosed by refusal, not deferral.** The institutional-foreclosure spirit of [ADR 0032](0032-personal-project-disposition.md) is preserved in full — the OSS pivot does not open the institutional path, it removes the cost-recovery framework that the institutional path was historically attractive against. Any future session proposing institutional licensing or grant pursuit must supersede this ADR with the same refusal-not-deferral discipline.

**(5) Cost-ceiling mechanism per [ADR 0029](0029-personal-financial-cost-ceiling.md) retires. The user's own Anthropic billing console is the per-user spend-cap surface; there is no aggregate-system cap (no aggregated Paideia API account exists under BYOK).** *Supersedes [ADR 0029](0029-personal-financial-cost-ceiling.md) in its entirety; [ADR 0029](0029-personal-financial-cost-ceiling.md) Status flips to `Superseded by ADR 0065`.* This is a mechanism change, not a justification-structure reframe. The pre-S-0012 framing of ADR 0029's cost-ceiling (bridge-to-grants until institutional capital arrived) was retracted at S-0012 by inline amendment because the mechanism survived — the same per-user / aggregate / telemetry / soft-wall apparatus continued to operate, just against a fixed personal subsidy budget rather than a transitional value. Under BYOK the apparatus itself does not exist: there is no Paideia-controlled API account against which a per-user cap could be enforced; there is no aggregated Paideia spend to telemetry; there is no soft-wall-at-cap behavior to wire because there is no Paideia-side cap. **The pedagogical principle the cost ceiling protected — degrade rather than terminate mid-concept-engagement — survives as a teaching discipline, absorbed into [ADR 0014](0014-sonnet-teaches-opus-reviews.md)'s role split.** A teaching-layer Sonnet that downshifts (deeper context window → shallower; two-hop retrieval → one-hop) when the conversation grows unwieldy is doing pedagogical work, not cost protection; the user's own Anthropic dashboard handles cost. OQ-WALL-BEHAVIOR closes in the same session (the cost-cap-degradation-ladder framing dies; the pedagogical-degradation discipline lives as part of [ADR 0014](0014-sonnet-teaches-opus-reviews.md)).

**(6) Polish is static, never dynamic. Maintenance is minimum-shape, deliberately.** *Preserved verbatim from [ADR 0035](0035-multi-platform-apple-expansion.md) commitment 6.* No social, no sharing, no leaderboards, no streaks, no push notifications beyond auth/billing, no community, no in-app messaging, no "what's new" surfaces. FAQ-only support; release cadence at builder discretion (4–8 week typical baseline). The threat shape under OSS+BYOK is now *user*-funnel-mechanic rather than *builder*-funnel-mechanic — engineering retention nudges that get users to spend more on their own Anthropic account is the same shape of corruption as engineering nudges that get users to renew a Paideia subscription, and the discipline that forbids the latter forbids the former.

**(7) Source is open under Apache 2.0. Copyright attribution to "The Paideia Project Contributors." The official Apple App Store binary is maintained under the original maintainer's personal Apple Developer account. Forks are permitted under the license; forks distributing through the App Store are encouraged to rebrand to avoid user confusion.** *New commitment.* Apache 2.0 is chosen for (i) permissive reuse, (ii) explicit patent grant covering the AI-pedagogy terrain where ambient patent risk is non-trivial, (iii) NOTICE-file attribution survival across forks, (iv) compatibility with App Store distribution (AGPL has documented tivoization-tension friction with the App Store that Apache 2.0 sidesteps), and (v) license alignment with the DeepTutor fork (also Apache 2.0 per [`product/docs/infrastructure.md`](../docs/infrastructure.md)). The "Paideia Project Contributors" attribution is generic — the builder's real name does not appear in the OSS-visible LICENSE or README. The official maintainer's Apple Developer account is the App Store identity. Apache 2.0 cannot prevent third-party forks from submitting competing App Store builds under their own developer accounts; the de-facto controls are App Store Connect ownership of the canonical listing plus the maintained domain pointer (paideia.app or equivalent). **Trademark registration on "Paideia" is deferred** — Greek-common-word distinctiveness is non-trivial, and the personal-project disposition does not justify proactive USPTO filing costs; file reactively if a competing fork appears in the wild and gains traction.

**(8) The user's Anthropic API key lives only on-device in iOS Keychain. No Paideia-controlled server receives the key, proxies the API, or observes the API exchange. Mastery computation continues server-side via structured `learner_events` derived client-side from prompt + response, consistent with the transcript-non-persistence commitment per [ADR 0026](0026-persistent-learner-storage-structural-not-substantive.md).** *New commitment.* iOS Keychain storage uses `kSecAttrAccessibleWhenUnlockedThisDeviceOnly` to prevent iCloud Keychain sync (the user should configure each device once; this is both a security clarity choice and a match for the Anthropic "one place to rotate" model). The key never appears in logs, never appears in crash reports (Sentry/Crashlytics/equivalent must scrub Authorization headers from any captured outgoing request), never appears in telemetry, never appears in error traces. Apple App Privacy questionnaire answer for the API credential: "data not collected." **The server's role under BYOK is unchanged in shape, narrower in scope** — it continues to own mastery-state computation via the event-sourced learner model per [ADR 0015](0015-event-sourced-learner-model.md), but it consumes only the structured `learner_events` the client derives from each engagement (per-engagement engagement-depth sub-signals per [ADR 0024](0024-engagement-depth-sub-signals-stored-raw.md), interaction type per the schema, exchange_summary as the structural-not-substantive JSONB shape per [ADR 0026](0026-persistent-learner-storage-structural-not-substantive.md)). The server never sees raw transcripts (already foreclosed by [ADR 0026](0026-persistent-learner-storage-structural-not-substantive.md)) and never sees the API request itself. The two reasons the prior architecture put mastery computation server-side (formula updates without client deploys; single source of truth across devices) both survive — they do not require the server to be in the API path.

**Success criterion: "An app I would pay for if it weren't mine."** *Preserved verbatim from [ADR 0035](0035-multi-platform-apple-expansion.md) and [ADR 0032](0032-personal-project-disposition.md).* The Phase 9 small-cohort cold-test (2–3 people who have not seen the project) remains the verification artifact. The cold-test now also exercises the in-app BYOK onboarding flow — does a freshman-defaults-calibrated user succeed at obtaining and entering an Anthropic API key without external help? This is the new builder-bias surface that did not exist under the prior disposition.

File a USPTO trademark application only if a competing fork appears in the App Store and gains traction; absorb the risk of confused-deputy fork releases as a low-probability cost given the personal-project disposition. The App Store Connect ownership of the canonical listing plus the maintained domain pointer (paideia.app or equivalent) provide the de-facto official-build signaling that suffices at this scale.

- **The corruption-vector test from [ADR 0032](0032-personal-project-disposition.md) generalizes to this ADR.** "Leave the option open to expand later" is the seed pattern future sessions are tested against. *Does this proposal preserve optionality whose value depends on a pedagogical or operational direction the builder no longer commits to?* The OSS+BYOK shift passes the test: it removes the cost-recovery surface entirely (no more pull toward retention/conversion mechanics that build value the builder doesn't want), and re-shapes the threat surface against the same discipline (user-funnel-mechanic forbidden by the same logic that forbade builder-funnel-mechanic). Future supersessions of this ADR are tested by the same generalization.

- **The refusal-not-deferral commitment is binding at the new boundary.** The most likely failure mode of a refusal-style commitment is silent re-litigation when growth makes the refused option attractive. The institutional/grant/acquisition foreclosure is preserved verbatim from [ADR 0032](0032-personal-project-disposition.md) commitment 4 (institutional half) and from [ADR 0035](0035-multi-platform-apple-expansion.md) — reopening any of these requires explicit supersession of this ADR with named context for why the disposition has changed.

- **Cancellation discipline survives unchanged.** Data-export per [ADR 0035](0035-multi-platform-apple-expansion.md) commitment 6 (preserved here as commitment 6); cascade-delete per [ADR 0031](0031-erasure-mechanism-and-individual-only-regime.md). Both are first-class UI primitives in Phase 9. The Apple Developer Program lapses if not renewed annually; the OSS source remains available regardless, so cancellation of the App Store binary does not destroy the artifact — a key property the OSS pivot strengthens.

- **This commitment does not rise to [`product/docs/MISSION.md`](../docs/MISSION.md)'s strong-working-commitments list.** Consistent with the same reasoning [ADR 0026](0026-persistent-learner-storage-structural-not-substantive.md), [ADR 0029](0029-personal-financial-cost-ceiling.md), [ADR 0031](0031-erasure-mechanism-and-individual-only-regime.md), [ADR 0032](0032-personal-project-disposition.md), and [ADR 0035](0035-multi-platform-apple-expansion.md) used: this is operating discipline (license + key-handling + distribution), not a pedagogical commitment. MISSION.md commitment 2's "operating discipline must not corrupt pedagogy" absorbs the OSS+BYOK shift the same way it absorbed the prior disposition revisions. The audience-facing pedagogical commitments survive in MISSION.md commitments 1–12 unchanged.

- **The MemPalace decision drawer for this ADR carries the conversational reasoning verbatim.** The S-0128 planning conversation that surfaced the OSS+BYOK direction shift — including the explicit "I have to make the repo public; I cannot ship my personal API key; both are forced together" reasoning and the four user-adjudicated decisions (attribution to "The Paideia Project Contributors"; keep `StarshipSuperjam/paideia` for now with milestone-bound migrate trigger; defer specific dollar pricing; defer trademark filing) — is recall-by-similarity content for future sessions facing analogous "should we open-source / should we BYOK / how to sequence dispositional shift" choices. The drawer is filed in this same session per the two-layer decision-recording discipline in [`CLAUDE.md`](../../CLAUDE.md).
