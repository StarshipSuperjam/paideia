# ADR 0095 — Phase 6 tool-stack: Postgres + JSONB + recursive CTEs confirmed; OSS Apache-2.0 + cost-priced posture raises the revisit bar

- **Status:** Accepted
- **Date:** 2026-05-18
- **Deciders:** S-0203 (PDG papers extraction pre-phase Session δ₂; sibling ADR 0096 lands the learning-outcome taxonomy in the same session)
- **Amends:** [ADR 0017](0017-postgres-recursive-ctes-over-owl-rdf.md) (Postgres + recursive CTEs over OWL/RDF) — same conclusion, narrower trigger, plus the OSS-license-compatibility constraint ADR 0017 did not carry pre-OSS-flip

## Context

[ADR 0094](0094-phase-6-scope.md) commits Phase 6 to expand to include Tier-A substrate redesign (Clusters 1-5) before SEP/embedding self-correction work commits node-table-dependent migrations. ADR 0094 premise P5 ("the tool-stack ADR does not constrain the scope choice") is named *with the explicit disposition that the Session δ₂ tool-stack ADR will need to either re-affirm Postgres+JSONB or change to Neo4j or RDF/Jena in concert with the self-correction master plan*. This is the Session δ₂ tool-stack ADR.

[`engine/build_readiness/pdg_papers_extraction/adversarial_review.md`](../../engine/build_readiness/pdg_papers_extraction/adversarial_review.md) E.10.3 (the adversarial-stance critique of synthesis Cluster 9 "Tool-stack evaluation: Postgres+JSONB+recursive CTEs vs Neo4j dual-layer") states:

> "C9 names 'Tool-stack evaluation: Postgres+JSONB+recursive CTEs vs Neo4j dual-layer. Per L3.15 finding, the threshold for graph-store evaluation is the introduction of analytics or trace-informed-revision workflows.' **The trigger criterion for evaluation ('introduction of analytics or trace-informed-revision workflows') is also the trigger for Phase 7+ work — meaning the evaluation never has a forcing function** [emphasis added]… the migration cost from Postgres to Neo4j is substantially higher (existing learner traces, existing provenance records, existing junction tables all need migration). The paper recommends Neo4j evaluation at design time, not after the Postgres schema has grown."

E.10.3's recommendation: "C9 should add a forcing function: the tool-stack evaluation (Postgres vs Neo4j) must occur **before any Tier A migration is applied to production**, not when Phase 7+ analytics arrive."

This ADR is that forcing-function settlement. It does **not** prolong the evaluation under a softer trigger — it adjudicates decisively for this Phase. The forcing function has fired; the answer is Postgres + JSONB + recursive CTEs; the alternative-class evaluation produces a foreclosure structure rather than a "look again later" deferral.

**The OSS posture is load-bearing for the foreclosure structure.** [ADR 0065](0065-oss-pivot-and-byok-disposition.md) (at S-0128) flipped the project to OSS Apache 2.0 + BYOK + solo-maintainer + cost-priced posture; [ADR 0093](0093-phase-6-product-trajectory-formalization.md) (at S-0202) re-affirmed for Phase 6 substrate work that OSS forks are not foreclosed and institutional pathways are foreclosed by refusal-not-deferral. Any tool-stack choice that requires forkers to obtain commercial database licensing (Neo4j Enterprise; AWS Neptune; Azure Cosmos Gremlin API; Memgraph BSL) is incompatible with "OSS forks not foreclosed" in practice — forkers at scale would have to either accept a feature-limited Community Edition with license-compatibility friction (Neo4j Community is GPLv3) or pay for Enterprise. Any tool-stack choice that imposes commercial-DB cost on the solo maintainer is incompatible with the cost-priced shipping commitment of ADR 0065 commitment 5. These constraints did not exist when ADR 0017 was authored (2026-04-09; pre-OSS-flip); they did not need to be named because the project's release surface was different. They exist now, and they raise the bar for revisiting Postgres+JSONB substantially.

**The pre-OSS-flip tool-stack reasoning still holds.** ADR 0017's three rejections of OWL/RDF, Neo4j-class dedicated graph stores, and "scale-driven graph-DB necessity" still stand on their original grounds:
- OWL class hierarchies reintroduce thinker-level aggregation (ADR 0008 violation).
- Dedicated graph stores introduce a second data system + second query language + cross-system joins for mastery-filtered traversal; mastery computation runs in Postgres natively (per [ADR 0085](0085-server-side-mastery-computation-confirmed.md) at S-0152).
- The performance case for a dedicated graph DB doesn't materialize until tens of thousands of nodes with complex multi-hop queries; Paideia's projected scale (hundreds to low thousands of nodes per subdomain) is comfortable on recursive CTEs.

What this ADR adds beyond ADR 0017:
1. **Decisive settlement** of E.10.3's forcing-function critique (the answer is Postgres; the decision is closed for this Phase, not deferred under a softer trigger).
2. **The OSS-license-compatibility constraint** as a new dimension that elevates the revisit bar (forecloses commercial and copyleft-friction graph stores at this posture level).
3. **The cost-priced-shipping constraint** as a foreclosing factor for any commercial-DB cost surface under ADR 0065 commitment 5.
4. **An explicit Apache-AGE pointer** as the only credible Postgres-internal property-graph revisit path if recursive-CTE performance ever becomes the actual bottleneck on Tier-A-redesigned schema.

[ADR 0084](../../engine/adr/0084-pushback-rule-extraction-step-for-high-stakes-decisions.md) applies — this is a **contract-shape change**: tool-stack settlement is the substrate every Phase 6 cluster-implementation ADR rests on, and the OSS-license-compatibility constraint propagates as a foreclosing factor for every future infrastructure choice (not just graph stores — analytics tools, search infrastructure, vector stores, etc.). The Load-bearing premises subsection below dogfoods the extraction step.

### Load-bearing premises

*(Per the [ADR 0084](../../engine/adr/0084-pushback-rule-extraction-step-for-high-stakes-decisions.md) extraction step.)*

1. **Tier-A schema (Clusters 1-5 per ADR 0094) is implementable in Postgres + JSONB + standard relational primitives.** *Falsifier:* a Tier-A cluster-implementation ADR (likely Cluster 3 with its `edge_outcome_necessity` junction or Cluster 5 with its misconception sub-graph linking edges to misconception nodes) discovers a structural shape that Postgres cannot model efficiently — e.g., variable-arity per-edge attribute sets whose JSONB representation costs more in query time than property-graph native traversal would. *Test status:* in-context partial test. Synthesis §Cluster 1 (per-edge contestability columns: `expert_confidence`, `trace_confidence`, `llm_confidence`, `disagreement_flag`, `provenance`, `counterexamples`, `version`, `review_status`, `last_reviewed`) is plain typed columns + JSONB for `counterexamples` — Postgres-native. §Cluster 2 (three-relation layering + 10 edge-types per layer) is a controlled-vocabulary CHECK constraint + a partition column — Postgres-native. §Cluster 3 (`edge_outcome_necessity` junction with `necessity_level`, `confidence_for_outcome`, `rationale_for_outcome`) is a standard FK-to-outcomes junction table — Postgres-native. §Cluster 4 (node schema redesign with `node_type` multi-valued + `disciplinary_domain` + `granularity` + `canonical_sources` JSONB + `misconceptions` JSONB) is text + JSONB array columns — Postgres-native (per-Q1 settlement this session, multi-valued `node_type` is a Postgres array column or JSONB array). §Cluster 5 (misconception sub-graph extending C2 edge layers + C4 `misconceptions` field) reuses C2's edge-type partition mechanism + C4's JSONB array — Postgres-native. **Premise verified for the four substrate-shape-bearing clusters.** Cluster 1 verification is unverifiable until the first Cluster 1 cluster-implementation ADR actually drafts the column set against Tier-A-redesigned `nodes`/`edges` tables; **named in Consequences as first-exercise readiness criterion T1-A** to close at the first Tier-A cluster-implementation ADR landing.
2. **Junction-table cardinality stays bounded within Postgres' comfortable operating range.** *Falsifier:* a Cluster 3 `edge_outcome_necessity` row count grows past the threshold where joins-during-traversal materially degrade learner-facing latency (~10k rows per edge would be a problem; ~10 rows per edge is comfortable). *Test status:* unverifiable in-context — depends on actual `learning_outcomes` table cardinality + per-edge necessity-fan-out. ADR 0096 (learning-outcome taxonomy ADR landing in this session) commits to a 3-level depth taxonomy + Decoding-the-Disciplines bottleneck moves — a bounded structured taxonomy whose row count grows linearly with discipline coverage, not combinatorially. Expected scale at Phase 6 close: 9 philosophy subdomains × ~3 learning outcomes per subdomain ≈ 27-50 rows; per-edge necessity fan-out averages ≤3 (most edges have one canonical outcome they support). Total junction-table size at Phase 6 close: ≤2000 rows for 533 edges. **Premise verified within the named Phase-6-projected scale**; **named in Consequences as first-exercise readiness criterion T1-B** to close at the first Cluster 3 migration landing with the actual row count measured.
3. **Recursive-CTE depth stays within ADR 0017's 8-10-hop comfort range.** *Falsifier:* a Tier-A redesign forces longer prerequisite chains (e.g., Cluster 4's `node_type` multi-valued redesign causes the SQA discipline to re-decompose long-chain concepts into shorter chains, paradoxically deepening the chains that remain). *Test status:* in-context partial test. The existing 380-node graph has prerequisite chains averaging ~5-7 hops per subdomain (from S-0122 production-audit findings). Tier-A redesign does not alter edge cardinality structurally — it redesigns the node schema and per-edge attribute set, not the topology. The chain-depth distribution is invariant under Cluster 1-5 changes. **Premise verified.** First-exercise readiness criterion not separately named — the Tier-A migrations themselves do not change topology; if Phase 7+ retrieval shape work (per ADR 0087 two-hop neighborhood) introduces deeper-chain query patterns, that ADR is the natural place to re-check.
4. **ADR 0017's OWL/class-hierarchy critique still holds against modern RDF/triplestore options.** *Falsifier:* a modern RDF/triplestore (Apache Jena, GraphDB Free, Stardog Community) offers OWL inference *without* forcing class-hierarchy reintroduction — e.g., by allowing instance-only graphs with SPARQL traversal and no inferred superclass machinery. *Test status:* in-context verified. The critique is structural: OWL's value-add over plain graph traversal is class-hierarchy inference (subclass-of reasoning, property domain/range inference, equivalent-class reasoning). An RDF store used *without* OWL inference is just a triple store + SPARQL — and Postgres + recursive CTEs already covers triple-store semantics with no second-system overhead. ADR 0008's concept-nodes-not-thinkers commitment forecloses the case where Paideia would benefit from class-hierarchy reasoning. **Premise verified.** ADR 0017's critique is reaffirmed without amendment.
5. **Apache 2.0-license-compatibility is binding under the ADR 0065 OSS-pivot posture.** *Falsifier:* [ADR 0065](0065-oss-pivot-and-byok-disposition.md) is superseded (the project flips back to closed-source distribution, or to a permissive-but-non-Apache-2.0 license, or to a copyleft license). *Test status:* in-context verified. ADR 0065 Status: Accepted; ADR 0093 commitment 2 re-affirms "OSS forks not foreclosed" specifically for Phase 6 substrate work; the OSS-flip tag `oss-flip-v1.0.0` landed at S-0130. The constraint is not "OSS forks must be easy" — it is "OSS forks must be possible at all," which forecloses dependencies that require commercial licensing to deploy at production scale. **Premise verified.** If ADR 0065 is later superseded by a closed-source flip ADR, this ADR's foreclosure structure weakens accordingly; the supersession would propagate through ADRs 0093, 0095, and forward.
6. **Solo-maintainer cost-priced shipping per [ADR 0065](0065-oss-pivot-and-byok-disposition.md) commitment 5 cannot absorb a commercial-DB cost surface.** *Falsifier:* ADR 0065 commitment 5 changes (the project pivots to a different cost model — e.g., venture-funded, donation-funded, institutional-sponsored — that can absorb commercial-DB costs). *Test status:* in-context verified. The current Supabase Postgres deployment cost is in the maintainer's named cost-ceiling; a commercial graph DB (Neo4j Enterprise on AuraDB cloud, e.g., starts at hundreds of dollars per month at production scale) would be a substantial multiple. The cost surface is not "a re-evaluation trigger" — it is a foreclosing constraint until the cost-priced posture changes. **Premise verified.**
7. **Apache AGE is the only credible Postgres-internal property-graph path if recursive-CTE performance later becomes the bottleneck.** *Falsifier:* Apache AGE goes unmaintained (community abandonment, version-incompatibility with current Postgres); OR a new Apache-2.0-or-MIT-licensed graph store reaches production maturity at Paideia's projected scale; OR PostgreSQL itself absorbs property-graph query primitives via a future extension or core feature (low probability within Phase 6+ horizon). *Test status:* in-context partial test. Apache AGE is currently active (last release 2024-11 with PG16 support; Apache 2.0 license; runs as a Postgres extension exposing Cypher-style queries against an internal graph storage layer that lives in the same Postgres database). Its primary use case is exactly the "Cypher convenience on Postgres-native data without standing up a second DB system" niche this premise names. The unknown is whether AGE's performance characteristics on Tier-A-redesigned schema improve recursive-CTE bottlenecks materially; that's a benchmark question for whenever the bottleneck actually arises. **Premise verified as the appropriate revisit target structure;** the empirical performance comparison defers to the time the bottleneck actually arises.
8. **E.10.3's "forcing function never fires" critique is satisfied by *decisively settling now*, not by deferring to a future re-evaluation trigger.** *Falsifier:* a downstream session re-opens the tool-stack decision without supersession-level evidence (i.e., without one of the four bar-clearing criteria below actually being met) — indicating the settlement was not load-bearing and the prior decision was still effectively in flight. *Test status:* in-context partial verification. The bar criteria (Decision section below) are structured to make "re-opening for a softer reason" mechanically detectable: if a session proposes Neo4j/Memgraph/Neptune/etc. *without* both demonstrated unfixable Postgres failure + Apache-2.0-compatible alternative + maintainer-cost-neutral evidence + amortizable-migration evidence, the proposal is structurally a supersession request that must clear ADR 0084's bar to land. **Premise verified at authoring;** falsification path is observable.

## Decision

**Phase 6 (and beyond, until the bar in this section is met) uses Postgres + JSONB + recursive CTEs as the graph substrate.** [ADR 0017](0017-postgres-recursive-ctes-over-owl-rdf.md) is re-affirmed with this amendment. The Tier-A migrations (per ADR 0094) commit to Postgres-native column types + JSONB array fields + standard junction tables; no second data system is introduced.

**Revisit bar: switching to a non-Postgres-internal graph substrate is justified only if all four of the following criteria hold.**

1. **Demonstrated unfixable Postgres scaling failure on production data.** Not a soft limit, not a benchmark hypothetical, not a "this might get slow at scale" projection. An actual production failure shape — measured query times exceeding the learner-facing latency budget — that has exhausted in-Postgres response: indexing strategies (B-tree, GIN, GiST on JSONB paths), materialized closure tables, recursive-CTE rewrites, partial precomputation, read-replica scaling, query-plan tuning. The Apache AGE extension (criterion 4) is one of the in-Postgres responses that must be exhausted first.
2. **The candidate alternative is Apache 2.0-compatible (or more permissive).** Downstream forkers per [ADR 0093](0093-phase-6-product-trajectory-formalization.md) commitment 2 must be able to deploy without commercial licensing or copyleft-friction.
3. **Maintainer-cost-neutral or favorable** vs the existing Supabase Postgres deployment, under the cost-priced-shipping commitment of [ADR 0065](0065-oss-pivot-and-byok-disposition.md) commitment 5.
4. **Migration cost amortizable** against the demonstrated benefit. The 16 existing Phase 5 migrations + learner_events + mastery_snapshots + provenance records + the Tier-A migrations themselves must be re-mapped under any tool-stack switch; the cost of this re-mapping must be smaller than the cost of staying on Postgres given the demonstrated failure in criterion 1.

**Effective foreclosures under this bar at authoring time:**

- **Neo4j Enterprise** — fails criteria 2 + 3 (commercial license; per-month cost).
- **Neo4j Community Edition** — fails criterion 2 under conservative interpretation (GPLv3; the Apache Software Foundation's published reading is that GPL is *category X — prohibited* for inclusion in Apache projects, though "depends on at runtime via network" arrangements are sometimes considered narrower in scope; the safe reading for an Apache-2.0 project distributing software is that GPLv3 backend dependencies are incompatible).
- **Memgraph Community Edition** — fails criterion 2 (BSL license with conversion-to-Apache-2.0-after-N-years; current state is non-compliant).
- **AWS Neptune, Azure Cosmos Gremlin API, GCP Spanner Graph** — fail criteria 2 + 3 (commercial cloud licensing; per-month cost).
- **Apache Jena / GraphDB Free / Stardog Community** — fail criterion 4 unless the OWL/class-hierarchy critique (premise 4) is overturned, because the only structural benefit over Postgres+CTEs at Paideia's scale would be OWL inference, which ADR 0008 forecloses.

**Effective admission under this bar at authoring time:**

- **Apache AGE (PostgreSQL extension exposing Cypher query syntax over an in-Postgres graph store)** — Apache 2.0 license; no second DB system; no maintainer-cost surface beyond the existing Supabase Postgres deployment. AGE is the only credible revisit target if recursive-CTE performance becomes the actual bottleneck. Adopting AGE would be a Phase-6+ amendment to this ADR, not a supersession, since it stays Postgres-internal.

**The decisions deferred by ADR 0094 premise P5 are settled.** The Phase 6 self-correction master plan (per ADRs 0085-0088) operates on the Tier-A-redesigned Postgres schema. SEP backfill + embedding work (per ADR 0086) operates against Postgres-stored embeddings (per-dim partition tables already committed). Two-hop retrieval (per ADR 0087) operates against Postgres edge tables. SEP chunk-resolver index (per ADR 0088) operates as a Postgres junction table. None of these need re-decomposition; ADR 0094 premise P5 is verified empirically by this settlement (Postgres-native; no tool-stack-change-forces-joint-redo).

## Alternatives Considered

### Postgres + JSONB + recursive CTEs (chosen)

- **What:** Continue ADR 0017's Postgres + recursive CTEs commitment with the OSS-license-compatibility constraint and Apache-AGE pointer added. No second DB system; no commercial licensing; no copyleft-friction. Tier-A migrations land Postgres-native.
- **Pros:** Zero migration cost for the 16 existing Phase 5 migrations + learner_events + mastery_snapshots + provenance records. Single DB system; operational simplicity per ADR 0017. Apache 2.0-compatible (Postgres license is permissive); ADR 0065 + ADR 0093 OSS posture preserved. Solo-maintainer cost-priced posture per ADR 0065 commitment 5 absorbs the existing Supabase cost surface without addition. ADR 0085 server-side-mastery + ADR 0086 per-dim embedding partitions + ADR 0087 two-hop retrieval + ADR 0088 SEP chunks all already commit to Postgres-native shape. Tier-A Clusters 1-5 substrate-implementable (premise 1 verification).
- **Cons:** Recursive-CTE performance is the bound on traversal depth. Long-chain queries (>10 hops) degrade; if Tier-A redesign deepens chains (premise 3 unfalsified), Apache AGE becomes the revisit target. Schema-evolution-as-data-grows still happens; large JSONB columns slow query plans if not indexed (GIN/GiST). Two-hop neighborhood retrieval (per ADR 0087) is implementable but query plans need attention as edge counts grow.
- **Rejected because:** not rejected — chosen.

### Neo4j (Community Edition or Enterprise)

- **What:** Migrate the graph layer to Neo4j; keep learner_events, mastery_snapshots, etc. in Postgres; bridge with cross-system joins or via application-layer dual writes. Use Cypher for traversal; eliminate recursive CTEs.
- **Pros:** Native property-graph traversal; multi-hop queries efficient; rich edge attributes supported as first-class concept; Cypher is concise for graph queries.
- **Cons (under Paideia's posture):**
  - **Neo4j Enterprise — commercial licensing (~hundreds/month at AuraDB scale).** Fails revisit-bar criterion 2 (Apache 2.0 compatibility — Neo4j Enterprise EULA is proprietary) and criterion 3 (maintainer-cost-neutrality — cost surface ADR 0065 commitment 5 cannot absorb).
  - **Neo4j Community Edition — GPLv3 license** creates Apache 2.0 compatibility friction for downstream forkers. The safe reading per Apache Software Foundation guidance is that GPL'd backend dependencies are incompatible with Apache 2.0 distribution. Fails revisit-bar criterion 2.
  - Migration cost from Postgres → Neo4j substantial (premise 1 cluster-set inventory + all existing schema). Fails revisit-bar criterion 4 absent demonstrated unfixable Postgres failure.
  - Mastery computation already settled to run in Postgres per ADR 0085; cross-system joins for mastery-filtered traversal would be the operational cost the entire ADR 0017 reasoning rejected.
- **Rejected because:** fails criteria 2 + 3 (Enterprise) or 2 + 4 (Community); under no current posture path can it clear the bar.

### Memgraph Community Edition

- **What:** Use Memgraph (Cypher-compatible, ~Neo4j-like API, in-memory-first with disk persistence) as the graph layer.
- **Pros:** Faster than Neo4j on many workloads; Cypher-compatible; community ecosystem.
- **Cons:** **BSL (Business Source License) — converts to Apache 2.0 after N years (typically 4); current state is non-compliant for Apache 2.0 distribution.** Fails revisit-bar criterion 2. Same cross-system-join concerns as Neo4j for ADR 0085 mastery computation. Smaller production deployment base than Neo4j.
- **Rejected because:** fails criterion 2 under BSL.

### Cloud graph DBs (AWS Neptune, Azure Cosmos Gremlin API, GCP Spanner Graph)

- **What:** Use a commercial cloud graph DB as the substrate.
- **Pros:** Managed-service operational ergonomics; scales without maintainer infrastructure work.
- **Cons:** Commercial licensing and per-month cost surface — fails revisit-bar criteria 2 + 3. Cloud-vendor lock-in is incompatible with "OSS forks not foreclosed" (per ADR 0093 commitment 2) — forkers must replicate the cloud deployment. Adds a third operational system (Postgres for relational + cloud-graph for graph + Supabase API surface).
- **Rejected because:** fails criteria 2 + 3; vendor lock-in additionally conflicts with ADR 0093 commitment 2.

### Apache Jena / GraphDB Free / Stardog Community (RDF triplestores)

- **What:** Use an RDF triplestore as the graph layer; express nodes and edges as triples; use SPARQL for traversal.
- **Pros:** Standards-based (W3C RDF + SPARQL); interoperability with semantic web tooling; permissive licenses (Apache 2.0 for Jena; LGPL for Stardog Community; GraphDB Free is closed-source-but-free).
- **Cons:** The structural value-add of an RDF store over Postgres + recursive CTEs is OWL inference (subclass-of reasoning, property domain/range, equivalent-class). ADR 0008's concept-nodes-not-thinkers commitment forecloses the case where Paideia would benefit from class-hierarchy reasoning (ADR 0017 reasoning preserved per premise 4). Without OWL inference, an RDF store is just a triple store + SPARQL — and Postgres + recursive CTEs already covers that shape with no second-system overhead. SPARQL is also a less-familiar query language for the solo maintainer.
- **Rejected because:** fails criterion 4 (migration cost not amortizable absent OWL-inference benefit, which ADR 0008 forecloses).

### Apache AGE (PostgreSQL extension)

- **What:** Install Apache AGE (https://age.apache.org/) as a Postgres extension; expose Cypher-style queries against an in-Postgres graph store; keep relational data + graph data in the same Postgres instance.
- **Pros:** Apache 2.0 license — meets revisit-bar criterion 2. No second DB system; no maintainer-cost surface beyond the existing Supabase Postgres deployment — meets criterion 3. Cypher convenience for graph queries; can interoperate with regular Postgres tables in the same instance.
- **Cons:** Adds an extension surface (version compatibility with PG16; depends on the AGE project's continued maintenance). Performance characteristics on Tier-A-redesigned schema are not measured; whether AGE materially outperforms recursive CTEs at Paideia's projected scale is an open empirical question. Supabase's hosted Postgres may or may not support custom extensions out-of-the-box; verifying AGE availability on Supabase is itself a small project.
- **Rejected at authoring time because:** **criterion 1 not met** — Postgres + recursive CTEs has not demonstrated unfixable scaling failure on production data; the in-Postgres response (indexing, closure tables, CTE rewrites) has not been exhausted. **Named explicitly as the only credible revisit target** if criterion 1 ever fires; the alternative-class table above is non-exhaustive only with respect to permissive-licensed extensions of this shape. Adopting AGE later would be an amendment to this ADR (still Postgres-internal), not a supersession.
