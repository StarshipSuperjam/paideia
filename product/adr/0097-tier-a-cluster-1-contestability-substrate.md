# ADR 0097 — Tier-A Cluster 1: Contestability substrate (per-edge confidence + provenance + counterexamples + version)

- **Status:** Accepted
- **Date:** 2026-05-18
- **Deciders:** S-0207 (first Tier-A cluster-implementation ADR per [ADR 0094](0094-phase-6-scope.md) Phase 6 scope expansion; dependency order C1 → C2 → C4 → C3 → C5)

## Context

[ADR 0094](0094-phase-6-scope.md) expanded Phase 6 to include Tier-A substrate redesign (Clusters 1-5) before the SEP/embedding self-correction work commits node-table-dependent migrations. [`engine/build_readiness/pdg_papers_extraction/synthesis.md`](../../engine/build_readiness/pdg_papers_extraction/synthesis.md) §Cluster 1 (lines 48-78) names contestability as **atomic, not piecemeal**: confidence (L1.4 three-source) + provenance (L1.5 five-field) + counterexamples (L1.7 array) + version history (L1.14) form a single governance substrate. Three independent lens-sweeps converge on this finding (synthesis top-level finding 3). Adding any one without the others produces unmoored data: confidence without provenance is an unjustified number; provenance without counterexamples is a one-sided record; either without version history loses the audit trail.

The current `public.edges` table at [`product/seed-graph/migrations/0003_edges.sql`](../seed-graph/migrations/0003_edges.sql) already carries two semantically-overlapping columns on 533 rows (516 `pedagogical_prerequisite` + 17 `historical_influence`):

- `confidence REAL CHECK (BETWEEN 0 AND 1) DEFAULT 1.0` — single-source confidence; synthesis requires three-source (expert + trace + llm).
- `provenance TEXT DEFAULT 'human'` — single-token marker; synthesis requires JSONB object with five named keys (`source_text` + `course_context` + `version` + `reviewer` + `rationale`).

The overlap forces an adjudication: rename + ALTER + backfill (preserve the 533-row data; converge on the synthesis-target shape) vs. coexist (add new columns alongside; leave Cluster 2/3 ADRs to break the tie) vs. drop + replace (clean target shape; lose the human-marked provenance signal on existing edges). The plan-mode adjudication for this session settled on ALTER + backfill — no data loss; clean target shape; no back-compat technical debt for downstream cluster ADRs.

Cluster 1 lands the first Tier-A cluster-implementation ADR per [ADR 0094](0094-phase-6-scope.md) dependency order. Its landing closes two ADR-level T1-A premises:

- **ADR 0094 T1-A** (premise 2 — "Tier-A implementability without re-decomposition"): closes when Cluster 1 either lands without scope re-shape (premise verified) or surfaces re-decomposition forcing amendment of ADR 0094.
- **ADR 0095 T1-A** (premise 1 — "Tier-A schema implementability in Postgres+JSONB"): closes when Cluster 1 either lands without Postgres-shape friction (premise verified) or surfaces a structural shape that Postgres cannot model efficiently, forcing amendment of [ADR 0095](0095-phase-6-tool-stack-postgres-jsonb-confirmed-with-oss-revisit-bar.md).

ADR 0094 T1-B already closed verified at S-0203 via D6 settlement; ADR 0094 T2-A (premise 6) closes later at the Phase 6 embedding-evaluation step per [ADR 0086](0086-model-agnostic-embedding-storage-architecture.md). ADR 0095 T1-B (premise 2 — junction-table cardinality bounded) closes at Cluster 3 migration landing.

[ADR 0084](../../engine/adr/0084-pushback-rule-extraction-step-for-high-stakes-decisions.md) applies — this is a **contract-shape change** under the ADR 0084 trigger classes (the per-edge contestability schema commits a contract that all downstream Phase 6 Tier-A cluster ADRs + the eventual Phase 7+ teaching-layer work + the eventual `graph_versions` table will build against). The Load-bearing premises subsection below dogfoods the extraction step.

[ADR 0053](../../engine/adr/0053-mechanism-first-exercise-gate.md) does NOT apply — this ADR authors schema, not a cross-cutting mechanism. No new wrapper around a harness surface; no novel cross-session protocol; trigger criteria #1-#4 do not fire. The first-exercise T1/T2 criteria below are premise-disposition closures specific to this ADR's load-bearing premises, distinct from ADR 0053's mechanism gate (same pattern as ADR 0094 / ADR 0095).

### Load-bearing premises

*(Per the [ADR 0084](../../engine/adr/0084-pushback-rule-extraction-step-for-high-stakes-decisions.md) extraction step. This ADR triggers under the "contract-shape change" class — the per-edge contestability columns commit a schema contract that all five Tier-A cluster ADRs + the Phase 7+ teaching layer + the deferred `graph_versions` table will build against.)*

1. **Contestability-as-atomic** — confidence (expert + trace + llm) + provenance (5-field JSONB) + counterexamples (JSONB array) + version + review_status + last_reviewed form a single governance substrate. Adding any subset produces unmoored data. *Falsifier:* a Cluster 2/3/4/5 cluster-implementation ADR discovers that a subset of these fields is sufficient for its purpose (e.g., the goal-relative parameterization junction table in Cluster 3 references only `expert_confidence` without using `counterexamples` or `provenance`, and the missing fields turn out to never be consulted by downstream code). *Test status:* in-context verified by synthesis corpus — the Substrate lens-sweep cross-cutting observation 1, the Adversarial lens-sweep cross-cutting observation 1, and the Evaluation lens-sweep converge independently on the atomic-substrate finding (synthesis top-level finding 3). **Premise verified.** Future cluster ADRs that consume only a subset of these fields do not falsify the premise — they consume *part of* the atomic substrate; the substrate's atomicity is about authoring-time inclusion, not query-time consumption.

2. **Three-source confidence model adequacy** — `expert_confidence` + `trace_confidence` + `llm_confidence` cover the three authority sources from which an edge claim derives. *Falsifier:* future edges require a fourth source of confidence not covered by the three (e.g., `peer_confidence` from crowdsourced validation; `automated_provenance_score` from a contradiction-detection pipeline). *Test status:* in-context partial. The three-source model is paper-2 L82 verbatim. The substrate lens-sweep enumerated no fourth source; the adversarial lens-sweep enumerated no fourth source. **Named in Consequences as a known assumption with Tier-2 readiness criterion**: if a Phase 7+ teaching-layer ADR or Phase 8+ evaluation-harness ADR proposes a fourth confidence source, ADR 0097 is amended (or superseded) to add the column. The schema is forward-compatible with extension (REAL columns + the disagreement_flag computation generalizes across N sources).

3. **Symmetric REAL representation across all three confidence sources** — modeling `expert_confidence` as REAL 0-1 (matching `trace_confidence` and `llm_confidence`) preserves mechanical disagreement_flag computability without losing the project's enum-band convention (low=0-0.4, medium=0.4-0.7, high=0.7-1.0 mapped at the UI/reporting layer). *Falsifier:* the enum-band-to-REAL mapping turns out to be lossy in a way that downstream consumers depend on (e.g., a UI surface that distinguishes "expert categorically said high" from "expert numerically marked 0.85"). *Test status:* in-context partial. The synthesis text frames `expert_confidence` as enum (low/medium/high) but `trace_confidence` and `llm_confidence` as REAL 0-1; the asymmetric mix forces an enum→numeric mapping anywhere disagreement_flag is computed. The plan-mode adjudication settled on symmetric REAL — the enum band remains as a UI/reporting convention, not a storage constraint. **Premise verified pending downstream consumer revisit**: if a Phase 7+ ADR needs the categorical-vs-numerical distinction at storage layer, this ADR can be amended to add an `expert_confidence_band` enum column alongside the REAL column without breaking existing consumers.

4. **Provenance five-field schema adequacy** — `{source_text, course_context, version, reviewer, rationale}` captures sufficient justification for future auditing and contestation. *Falsifier:* an audit reveals that a key justification dimension is missing — e.g., `epistemic_status: empirical_vs_normative` distinction; or `consultation_chain` listing prior reviewers' positions. *Test status:* in-context partial. The five fields are from paper 2 L83 verbatim; the Decoding-the-Disciplines authoring discipline cited by Tier B Cluster 6 has not yet been operationalized so the audit-time prompt set is not yet known. **Named in Consequences as a known assumption with Tier-2 readiness criterion**: the Phase 6.5+ authoring-discipline ADR (Cluster 6) or the Phase 8+ evaluation-harness ADR will be the first natural empirical test of provenance-field adequacy; if a missing dimension surfaces, ADR 0097 is amended. The JSONB shape is forward-compatible — adding fields requires only an ALTER TABLE on the CHECK constraint, no data migration.

5. **Counterexample array semantics** — each entry `{description: TEXT, evidence_link?: TEXT}` is sufficient for capturing the kind of exceptions-to-the-edge that the papers name (paper 1 L116-117 schema table; paper 2 L81 case studies). *Falsifier:* counterexamples require additional structure — e.g., `refutation_strength` enum, `alternative_conclusion` text, `dependency_on_outcome_id` FK (linking the exception to a specific learning_outcome from Cluster 3). *Test status:* in-context partial. The papers give worked examples (Kant/phenomenology, Saussure/Derrida) but no formal counterexample schema. **Named in Consequences as a known assumption with Tier-2 readiness criterion**: Cluster 5 (Misconception sub-graph encoding, per ADR 0094 dependency order C5) is the natural empirical test — if Cluster 5's misconception-link semantics need counterexample fields beyond `description + evidence_link`, ADR 0097 is amended. The JSONB array shape is forward-compatible.

6. **disagreement_flag computation specification, threshold value deferred** — `disagreement_flag = (max(non_null_confidences) - min(non_null_confidences)) ≥ DISAGREEMENT_THRESHOLD` is the canonical computation; the threshold value `DISAGREEMENT_THRESHOLD` is **deliberately not specified in this ADR**. *Falsifier:* the computation shape turns out wrong (e.g., disagreement is better expressed as pairwise rather than range-based; or as variance rather than max-min spread). *Test status:* in-context partial. Range-based max-min is the simplest and most defensible default; pairwise (∃(i,j) such that |c_i - c_j| ≥ θ) is equivalent for the 3-source case when θ is the same. Variance requires more data points to be meaningful and degenerates at N=3. **The threshold value itself is unverifiable in-context** — calibration requires learner trace data (SQA validation per [ADR 0022](../../engine/adr/0022-periodic-project-health-checks.md) discipline extended to per-edge contestability per Cluster 7's predictive-validity work) which does not yet exist. **Named in Consequences as a Tier-1 readiness criterion** (ADR 0097 T1-B): the threshold value is committed in a follow-up amendment to this ADR (or a sibling ADR) when learner trace data accumulates; until then, `disagreement_flag` remains FALSE for all existing edges (computed-set field; no rows trigger the threshold because no edges have `trace_confidence` or `llm_confidence` populated yet).

## Decision

**Land the per-edge contestability substrate as the first Tier-A cluster migration**, structured as nine column-level changes to `public.edges` adjudicating two ALTER overlaps and seven new column additions. The migration file at [`product/seed-graph/migrations/0120_edges_contestability_substrate.sql`](../seed-graph/migrations/0120_edges_contestability_substrate.sql) (Phase 6 self-correction prefix range per [`ROUTING.md`](../seed-graph/migrations/ROUTING.md) — Tier-A substrate cluster work claims the Phase-6-self-correction range slots) executes the changes atomically in a BEGIN/COMMIT transaction with full contract header + Postcondition-Assertions block per [ADR 0055](../../engine/adr/0055-apply-migration-wrapping-against-production-reads-gate.md) Layer 2.5 grammar. Paired rollback at [`product/seed-graph/migrations/0120_edges_contestability_substrate_rollback.sql`](../seed-graph/migrations/0120_edges_contestability_substrate_rollback.sql) per [`engine/operations/migration-discipline.md`](../../engine/operations/migration-discipline.md) "Rollback authorship".

### Column-level commitments

**Existing columns — ALTER with backfill:**

- `confidence REAL` (existing) → `expert_confidence REAL`: column renamed via `ALTER TABLE ... RENAME COLUMN`. The 533-row data is preserved as-is — every existing edge carries `expert_confidence = 1.0` (the prior `confidence` default). CHECK constraint `expert_confidence BETWEEN 0 AND 1` preserved (constraint name updated). Forward-compatible with future expert-driven calibration of the existing edges.

- `provenance TEXT` (existing) → `provenance JSONB`: column type altered with USING clause that backfills the 533 existing rows. Every existing edge carries `provenance = '{"source_text": null, "course_context": null, "version": 1, "reviewer": "human", "rationale": null}'::jsonb` post-backfill (the `reviewer: 'human'` token preserves the prior `provenance = 'human'` signal; `source_text` / `course_context` / `rationale` are NULL pending Cluster 7 SQA backfill). CHECK constraint enforces `jsonb_typeof(provenance) = 'object'` AND all five required keys present. JSONB shape is forward-compatible per premise 4.

**New columns — ADD COLUMN:**

- `trace_confidence REAL NULL CHECK (trace_confidence IS NULL OR (trace_confidence BETWEEN 0 AND 1))`. NULL until learner trace data exists; populated by Cluster 7's SQA predictive-validity pipeline.
- `llm_confidence REAL NULL CHECK (llm_confidence IS NULL OR (llm_confidence BETWEEN 0 AND 1))`. NULL except for LLM-proposed edges (no such edges exist at this ADR's landing; populated by Phase 6.5+ LLM-assisted edge-authoring pipeline if/when it ships).
- `disagreement_flag BOOLEAN NOT NULL DEFAULT FALSE`. Computed-set field — never user-set; populated by a triggered or scheduled job applying the computation in premise 6 once `DISAGREEMENT_THRESHOLD` is calibrated. All 533 existing rows default to FALSE (no `trace_confidence` or `llm_confidence` populated yet; computation collapses to single-value range = 0 < any positive threshold).
- `counterexamples JSONB NOT NULL DEFAULT '[]'::jsonb CHECK (jsonb_typeof(counterexamples) = 'array')`. Empty array for all 533 existing rows. Each entry shape `{description: TEXT, evidence_link?: TEXT}` per premise 5 (JSONB array element shape not CHECK-enforced — JSONB per-element type-check across an array is unwieldy in plain CHECK; per-element shape is documented in the ADR + verified at application layer).
- `version INTEGER NOT NULL DEFAULT 1 CHECK (version >= 1)`. Monotonically increasing per edge. All 533 existing rows default to 1; the eventual `graph_versions` table (deferred — see below) will link version IDs to active edge/node sets per course/cohort.
- `review_status TEXT NOT NULL DEFAULT 'provisional' CHECK (review_status IN ('approved', 'pending_review', 'rejected', 'provisional'))`. All 533 existing rows default to `'provisional'` — they retype to `soft_prerequisite` by Cluster 2 ADR (per [ADR 0094](0094-phase-6-scope.md) Decision §1 dependency note + synthesis line 109) pending SQA validation; `'provisional'` is the matching review-lifecycle posture. Cluster 2 ADR coordinates this default.
- `last_reviewed TIMESTAMPTZ NULL`. NULL until first review; populated by Cluster 7's SQA pipeline or by manual reviewer action.

**Existing columns unchanged**: `id` (PK), `source_id` + `target_id` (FK to nodes), `edge_type` (TEXT, intentionally unconstrained per architecture.md:182), `weight` (REAL, CHECK 0-1), `evidence` (TEXT, nullable — distinct from `counterexamples`: `evidence` supports the edge claim; `counterexamples` records exceptions), `graph_version_added` (INTEGER, distinct from new `version` column — `graph_version_added` records the migration-session graph-version at insertion time per the [`ROUTING.md`](../seed-graph/migrations/ROUTING.md) graph_version increment contract; `version` is the per-edge revision counter), `created_at` + `updated_at`. UNIQUE(source_id, target_id, edge_type) preserved. RLS policy `edges_authenticated_read` preserved. The 533-row count is invariant — this is a pure schema change, not a seed.

### Explicit deferrals

- **`graph_versions` table** (synthesis line 68) deferred to a follow-up sibling Tier-A migration, likely paired with Cluster 3 (Goal-relative parameterization) when the `learning_outcomes` table lands — cohort/course-scoping for versioning is goal-relative. The per-edge `version` column is Cluster 1's commitment; the cohort-scoping table is Cluster 3+'s commitment. Documented as a forward pointer in Consequences.

- **`DISAGREEMENT_THRESHOLD` numeric value** deferred per premise 6 — calibrated post-SQA when learner trace data exists. Named as ADR 0097 T1-B. The computation is specified; the value is not.

- **Validator soft-warn for unguarded high-confidence claims** (synthesis line 70 — "any edge with `expert_confidence: high` AND empty `counterexamples` AND empty `provenance.rationale` is flagged as an unguarded high-confidence claim") is committed as a Consequences-deliverable (see below) but implementation defers to a follow-up issue. Rationale: the soft-warn requires real Cluster 2 + Cluster 3 context to be useful — today all 533 existing edges have `expert_confidence = 1.0` (high band) AND empty counterexamples AND empty `provenance.rationale`, so the soft-warn would fire on 533 rows immediately, drowning the signal. Cluster 2 retypes 516 to `soft_prerequisite` with `expert_confidence: low` defaults; after Cluster 2 the soft-warn is meaningful (it flags genuinely-unguarded high-confidence edges, not the entire seed corpus). Follow-up Issue filed at session shutdown.

## Alternatives Considered

### ALTER + backfill on existing `confidence` and `provenance` columns *(chosen)*

- **What:** Rename `confidence` → `expert_confidence`; ALTER `provenance` TEXT → JSONB with USING-clause backfill assigning `{"reviewer": "human", "rationale": null, ...}` to all 533 existing rows. Add 7 new columns alongside.
- **Pros:** Preserves the existing data signal (the 1.0 prior `confidence` becomes 1.0 `expert_confidence`; the `'human'` prior `provenance` token becomes `provenance.reviewer = 'human'`). Converges on the synthesis-target shape with zero data loss. Downstream cluster ADRs (Cluster 2 retyping; Cluster 3 junction table; Cluster 7 SQA validation) consume the target shape directly without back-compat shims. Single source of truth per dimension (no two semantically-overlapping columns to choose between).
- **Cons:** Larger migration body (rename + type-alter + backfill + 7 ADD COLUMNs in one transaction). Rollback complexity higher (rollback must restore both pre-state column types AND the data the type-conversion losslessly mapped — see paired `0120_edges_contestability_substrate_rollback.sql`). Type-alter with USING clause requires all 533 rows to validate against the new CHECK constraint on first apply (caught by Layer 2.5 postcondition-assertions).
- **Rejected because:** not rejected — chosen.

### Coexist (add new columns alongside existing `confidence` and `provenance`)

- **What:** Keep `confidence REAL` and `provenance TEXT` as-is. Add `expert_confidence REAL` + `provenance_meta JSONB` (or similar) + the 6 other new columns alongside. Defer the convergence to a follow-up Cluster 2/3 ADR.
- **Pros:** Smaller initial migration. Zero risk of backfill USING-clause edge cases. No rollback complexity — drop the new columns and the old shape is unchanged.
- **Cons:** Two semantically-overlapping columns per dimension (confidence + expert_confidence; provenance + provenance_meta) require every downstream consumer to choose between them — Cluster 2's retyping ADR would have to specify "use expert_confidence not confidence for the soft_prerequisite default"; Cluster 7's SQA validator would have to specify which column gates the validation. The choice itself is the back-compat technical debt the ALTER option avoids. The "let Cluster 2/3 break the tie" disposition defers a decision that has no good resolution — eventually someone DROPs the old column with backfill, which is the same migration the ALTER option does now but with one extra round-trip.
- **Rejected because:** the convergence work is the same in both options — ALTER pays it now; coexist pays it later with additional decision-deferral cost. The risk of USING-clause edge cases is bounded (533 rows; deterministic mapping; verifiable by Layer 2.5 assertions). The "smaller initial migration" benefit is illusory — the second migration is larger because it must consume both old and new columns to do the convergence.

### Drop + replace (DROP existing `confidence` and `provenance`; ADD new columns with no backfill)

- **What:** DROP existing `confidence` and `provenance`; ADD `expert_confidence` REAL + `provenance` JSONB + 6 others with their defaults; the 533 existing rows get `expert_confidence = 1.0` (new column default) and `provenance = '{}'::jsonb` (new column default).
- **Pros:** Cleanest target shape; no USING-clause complexity; the rollback is symmetric (drop the new columns, re-add the old with defaults).
- **Cons:** Loses the `'human'` provenance token signal on the 533 existing edges — they become `provenance = {}`, which is the same as edges whose provenance was never authored. The signal that these edges were human-authored is destroyed. Cluster 7's SQA validation downstream cannot distinguish "originally human-authored" from "no provenance ever supplied" without this signal.
- **Rejected because:** data signal loss is permanent — once dropped, the prior `'human'` provenance cannot be recovered. The 533 existing edges are the foundation of the seed graph; their provenance signal is load-bearing for any downstream auditing or contestability analysis. The cleanest-target-shape benefit does not outweigh the data-loss cost.

### Synthesis-verbatim asymmetric confidence types (expert ENUM, others REAL)

- **What:** Honor synthesis line 58-60 literally — `expert_confidence ENUM (low, medium, high)` + `trace_confidence REAL` + `llm_confidence REAL`. Implement disagreement_flag via an enum→numeric mapping function (low=0.3, medium=0.6, high=0.9).
- **Pros:** Matches synthesis text verbatim. Categorical expert-judgment band is explicit at the storage layer (the UI/reporting convention is enforced by the column type, not by application-layer mapping).
- **Cons:** Asymmetric storage forces an enum→numeric mapping anywhere disagreement_flag computes — the mapping is application-layer code; the mapping values (low=0.3, medium=0.6, high=0.9) are an arbitrary design choice not in the synthesis text and themselves load-bearing without further deliberation. Storing expert_confidence as enum prevents experts from expressing a calibrated mid-band judgment (e.g., 0.85 as distinct from 0.7 within "high"); the loss-of-precision is consequential when downstream consumers want continuous discrimination. The disagreement_flag computation also has to handle the case where all three sources are populated but expert is enum-mapped to one of three discrete values — disagreement is structurally larger than the underlying judgment because the enum mapping has coarser granularity than the others.
- **Rejected because:** symmetric REAL preserves the synthesis intent without the precision loss; the enum band remains as a UI/reporting convention (low=0-0.4, medium=0.4-0.7, high=0.7-1.0 at the presentation layer), which is the right layer for categorical-band presentation. The premise-3 analysis in Load-bearing premises above expands on this.

### Defer Cluster 1 entirely; start with Cluster 2 (edge-type taxonomy)

- **What:** Land Cluster 2 ADR + migration first; let Cluster 1 land second after Cluster 2's edge-type retyping clarifies the per-edge contestability requirements.
- **Pros:** Cluster 2 retypes 516 edges from `pedagogical_prerequisite` to `soft_prerequisite` (per synthesis line 101) — that retyping naturally exercises the per-edge `review_status` column Cluster 1 would add; testing the Cluster 1 schema against real retyped data might surface design issues Cluster 1 alone cannot.
- **Rejected because:** [ADR 0094](0094-phase-6-scope.md) Decision §1 explicitly commits the dependency order 1 → 2 → 4 → 3 → 5; synthesis line 109 explicitly notes "the edge migration [Cluster 2] is cleaner if Cluster 1's per-edge confidence + provenance columns exist first, so each retyped edge can be authored with `provisional` confidence pending SQA re-validation." Reversing the order violates the dependency rationale ADR 0094 settled. The "test Cluster 1 schema against real retyped data" pro is achievable post-landing — Cluster 2's retyping will exercise the Cluster 1 schema in production, and if design issues surface, Cluster 1 ADR can be amended (or superseded) with the empirical signal.
