>
> Stays at root per [ADR 0037](engine/adr/0037-engine-product-wall-and-changelog-rename.md) edge-case (c) — Claude Code expects it there. Content is engine-side; the wall the ADR commits to is conceptual, not a physical-purity test.

## Startup ceremony

Every session begins by reading these in order:

1. **`engine/STATE.md`** — current phase, last-closed session, next-session work item, infrastructure pointers.
2. **`engine/session/register_state.json`** — counter state. If `next_id`'s 4-digit suffix mod 30 == 0, the health-check cadence trigger fires (see `engine/operations/health-check.md`). The `SessionStart` hook (`engine/tools/hooks/session-start.sh` per [ADR 0043](engine/adr/0043-hook-architecture.md)) emits the cadence surface from the harness side regardless of how the session is launched, so steps 1–2 are also mechanically backstopped.
3. **MemPalace** — query `mempalace_search` with terms derived from the next-session work item. Surface anything relevant the user previously named.
4. **Files referenced by `engine/STATE.md` and `ROADMAP.md`** for the current work.

`/start-engine` automates this and claims the next slot. The slash command's documented procedure routes through the `session-build-lifecycle` Skill (per [ADR 0044](engine/adr/0044-skill-conversion-recipe-vs-reference.md)). See `.claude/commands/start-engine.md` and `.claude/skills/session-build-lifecycle/SKILL.md`.

A fresh session reading only this file learns the rules but not the procedures. The smallest set of operations docs to read on cold start, in order:

## Three session modes

- **Default — exploration.** No project file edits to tracked files. No commits. No slot claim. Sketch in conversation. MemPalace captures with the `exploration` tag (via the stop/precompact hooks in `.claude/settings.json`). When discussion converges on something worth committing, offer `/start-engine`.
- **Build — `/start-engine`.** Eager-claims the next slot, allows commits and project file edits, runs the shutdown sequence at close. Canonical invocation routes through three user-defined Skills under `.claude/skills/` (per [ADR 0044](engine/adr/0044-skill-conversion-recipe-vs-reference.md)):
  - `session-build-lifecycle` — boot procedure invoked at session start (Layer 1 doc: `engine/operations/session-build-lifecycle.md`).
  - `session-shutdown-sequence` — close procedure invoked at session end (Layer 1 doc: `engine/operations/session-shutdown-sequence.md`).
  - `build-readiness-gate` — gate procedure invoked before substantive build sessions (Layer 1 doc: `engine/operations/build-readiness-gate.md`).

  Routine mode (below) adds a fourth Skill `routine-mode-lifecycle` invoked by `/start-routine`. All four Skills carry `disable-model-invocation: true` — the AI invokes deliberately rather than auto-firing on description match. The Layer 1 ops docs remain the source-of-truth; updates flow doc → skill, never the reverse.

- **Routine — `/start-routine`, unattended cadence-driven** (per [ADR 0051](engine/adr/0051-routine-mode-and-engine-loop.md)). A Claude Code Routine fires sessions on a cadence; each fired session invokes `/start-routine`, which reads [`engine/session/auto_target.json`](engine/session/auto_target.json), picks the next eligible task, runs plan-then-scope-check, executes or exits gracefully without claiming a slot. Layer 1 doc: [`engine/operations/routine-mode-operations.md`](engine/operations/routine-mode-operations.md). Skill: [`routine-mode-lifecycle`](.claude/skills/routine-mode-lifecycle/SKILL.md). Architecturally distinct from `/start-engine` — they share only the eager-claim ritual; every other concern (boot procedure, scope rules, commit posture, permission model, shutdown logic) is routine-specific.

  **Routine-mode posture (load-bearing):**
  - **`scope_lock` is invariant on task-deliverable paths.** The active task's `scope_lock.allowed_paths` is the cap. Pre-commit hook hard-fails any commit touching paths outside this set ∪ the operational allowlist.
  - **Operational allowlist always permitted.** Files the session apparatus runs on (`engine/session/current.json`, `current_plan.md`, `auto_target.json` (status fields only), `engine/session/archive/S-*.json`, `register_state.json`, `ENGINE_LOG.md`, `HANDOFF.md`) commit freely regardless of active task.
  - **`gh issue create` is explicitly authorized** for in-band discoveries (bugs, tech-debt, cleanup, enhancements) outside the current task's scope. No tracked files touched, scope_lock irrelevant. Continue with assigned task.
  - **HANDOFF additions are explicitly authorized** for genuine blockers, scope-expansion-needed, and decision-required signals. Existing `audit_handoff_dispositions.py` audit (per ADR 0048) catches malformed entries.
  - **Master plan revisions ONLY via HANDOFF — never unilateral edits.** Pre-commit hook hard-fails any routine commit to `auto_target.json` that diffs keys other than `tasks[*].status` and `tasks[*].blocked_reason`. The user adjudicates revisions in interactive sessions.
  - **`routine_eager_claim_recovery.py` is explicitly authorized** (per [ADR 0082](engine/adr/0082-routine-boot-freshness-and-concurrency-defense.md)). The narrow `git reset --hard origin/main` it performs is mechanically verified to be a bounded eager-claim-race recovery (HEAD = exactly one commit ahead of origin/main, that commit's subject matches the eager-claim convention, origin/main HEAD also has an eager-claim for the same slot). Outside the verified shape the destructive-action refusal stands. Boot-freshness fast-forward (`routine_boot_freshness.py`) is also authorized — `git merge --ff-only` moves HEAD forward without merging or discarding anything.

  **Routine-mode runs in `Default` permission mode**, not `plan` mode — plan mode requires human ExitPlanMode approval that an unattended session cannot provide. The plan-then-scope-check (write `engine/session/current_plan.md`, then run `engine/tools/check_routine_scope.py --plan`) is the externalization-as-quality-gate equivalent, mechanically enforced rather than human-gated.

## Standing rules

### Pushback rule

The AI is pre-authorized to surface unnamed risks, hidden pitfalls, and unstated opportunities at the moment of noticing. Push back specifically — name the concern concretely. Apply equally to user proposals AND AI proposals (self-critique). The bar is "I see a specific thing you may not be seeing," not "I should challenge this on principle." Calibrate by mode: looser when the user is venting/exploring, tighter when proposing a commitment.

**Extraction step for high-stakes decision classes** (per [ADR 0084](engine/adr/0084-pushback-rule-extraction-step-for-high-stakes-decisions.md)). For ADRs in these classes — (1) cross-cutting mechanism authoring per [ADR 0053](engine/adr/0053-mechanism-first-exercise-gate.md) trigger #4 (Consequences span ≥5 surfaces), (2) supersession of a load-bearing prior decision, (3) posture-to-machinery conversion, (4) contract-shape change (structured-archive field per ADR 0042, validator phase budget per ADR 0063, ADR sequence per ADR 0037) — the AI runs an explicit extraction step **before** authoring the Decision section. At the bottom of the Context section, author a `### Load-bearing premises` subsection listing each premise the Decision rests on (one bullet), naming what would falsify each (one sentence), and running a cheap empirical test now for any premise that can be tested in-context. Premises that can only be verified post-landing are named in Consequences as known assumptions with a fallback (typically a first-exercise readiness note or a Tier-1 closure criterion). The pushback rule is reactive ("I see a specific thing"); the extraction step is the proactive complement that forces enumeration of premises BEFORE the author has the specific thing to surface — closing the gap that ADR 0054 (interactive-sessions-don't-trigger-gate, falsified at S-0137) and ADR 0063 (subprocess-probes-are-structural, falsified at S-0127) cost the project. Small targeted ADRs (typo fixes, doc rewrites, narrow validator extensions) do not trigger. Posture-not-machinery — no commit gate; inspectable post-hoc via ADR reading.

### Auto-mode interrupt criteria

While running in auto/build mode, do NOT pause and escalate to the user EXCEPT for:

- **Irreversible-with-unclear-path** — a decision propagates as irreversible structure across multiple downstream sessions AND the right direction is genuinely unclear.
- **Unsolvable** — multiple approaches tried, no viable path.
- **Destructive-action confirmation** — any `rm -rf`, `git reset --hard`, force-push, deleting tracked files in bulk. Confirm before executing.

Routine judgment calls during a session are session-internal and recorded in `engine/session/current.json`'s `outcome_summary`, not escalated.

### Budget guidance

- Substantive extraction work: target 60% context, hard cap 70%.
- Mechanical/procedural work: target 70%, hard cap 80%.

The percentages refer to the active context window (1M tokens for Claude Opus 4.7, the model the project currently runs under). Absolute thresholds: ~600K (substantive target), ~700K (substantive cap), ~700K (mechanical target), ~800K (mechanical cap). These are higher than pre-1M intuitions; a session at 30% load (~300K tokens) is comfortably within budget, not near the cap.

**Do not propose splitting, deferring, or re-scoping a session based on "this feels big" alone.** Read the actual percentage. The default is single-session full execution; splits require a concrete blocker (the percentage itself, an external constraint surfaced mid-work, a structural reason such as foundations-vs-mechanization that's named in the plan up front).

Worked examples:

- 20% load while planning a multi-file refactor: not a budget concern. Continue.
- 35% load mid-extraction with three large operations docs still to read: not yet a concern; estimate the additional load (each doc adds ~5-10% typically) and continue.
- 65% load mid-extraction with substantive new code yet to author: this is the moment to consider whether to halt at the next sensible boundary. Concrete cap, not "feels big."
- 75% load on mechanical procedural work (drift sweep, link audit): approaching the mechanical cap; either finish the immediate task and close, or queue the remainder as next-session work with a specific pointer.

If a session hits its cap mid-work, halt at the next sensible boundary, write `outcome_summary` with partial-completion notes, archive `engine/session/current.json` with `status: closed_partial`, push, and the next session picks up.

### End-state-quality first-pass

Author at production quality the first time. Don't write a draft and polish; don't write a placeholder and revisit. The cost of re-reading and re-revising a sloppy first pass exceeds the cost of slowing down to write it correctly. Particularly true for documents that downstream sessions will read cold — readability *is* the artifact's value.

Applies to newly-authored content. Cleanup sweeps that retrofit existing content to a newly-imposed contract — applying [ADR 0036](engine/adr/0036-expression-contract-for-inward-documents.md) to documents authored before the contract existed, for example — are exempt and belong in the new contract's own Consequences section.

### Default to fix-in-context

A bug found mid-session whose fix you understand defaults to **inline fix in this session**, not a HANDOFF.md entry. The deferral overhead — writing prose that names the bug, names the fix, names the proposed verification, plus the future session re-deriving the same details from cold — exceeds the inline-fix cost in nearly every case. Even when the bug is preexisting (not introduced by this session) and even when it's tangential to the session's named scope, the calculus holds: if you have the fix in context, apply it.

Three named exceptions admit deferral:

- **Substantial scope.** The fix touches load-bearing contracts (multiple ADRs, a posture rule, a cross-cutting refactor). Estimate honestly — most "feels substantial" reads are wrong.
- **Contract change required.** The fix mutates a public surface another file or session depends on; that mutation needs its own ADR-tracked deliberation.
- **Budget cap reached.** Per the budget-guidance percentages above, the session genuinely cannot fit the additional work without spilling.

If one of those three applies, **flag the user before deferring**: name the bug, name the fix, name which exception applies, ask whether to fix in context now or defer. Do not write the deferral artifact (HANDOFF entry or Issue) first.

**Where deferrals go (per [ADR 0048](engine/adr/0048-handoff-narrowing-and-github-issues-for-cross-session-deferrals.md)).** HANDOFF.md is reserved for *session-internal handoffs* — work the next session must pick up immediately because the closing session was halted at a sensible boundary, or because there's an in-flight recovery procedure, or because work-item ownership transfers atomically. Cross-session deferrals — bugs, tech-debt, cleanup, enhancements, doc work, open questions, health-check findings — route to GitHub Issues via `gh issue create` with appropriate labels (`bug`, `enhancement`, `tech-debt`, `cleanup`, `health-check-finding`, `upstream`, `documentation`, `question`, `priority:urgent`). The choice is governed by [`engine/operations/issue-discipline.md`](engine/operations/issue-discipline.md): if the next session must address this before its own scoped work, HANDOFF.md; otherwise, Issue. The default is Issue. Future sessions see the backlog count at every boot and can dedicate cleanup-batch sessions when the count crosses a threshold worth addressing.

Mechanically backstopped by [`engine/tools/audit_handoff_dispositions.py`](engine/tools/audit_handoff_dispositions.py): every new section added to HANDOFF.md during a session must carry a `**Disposition:**` line in one of five accepted forms (`fixed-in-session @ <SHA>`, `deferred-with-user-confirmation`, `out-of-scope`, `not-actionable`, `tracked-as-issue #<num>`). The audit hard-fails at session shutdown if any new section is missing or has an unrecognized disposition. The intent is to make the deferral choice explicit at authoring time; the audit catches lapses.

### Two-layer decision recording

Every settled decision lands in two places, serving different consumers:

- **ADR** — durable, structured, citable. The contract. Engine ADRs (about how the AI build apparatus works) live in `engine/adr/`; product ADRs (pedagogy, learner model, curriculum, schema, business) live in `product/adr/`. Partitioned at S-0024 per [ADR 0037](engine/adr/0037-engine-product-wall-and-changelog-rename.md).
- **MemPalace** (`decision` tag) — semantic memory, conversational form, recall-by-similarity. The story.

ADRs answer "what's settled?". MemPalace answers "have we considered X before?". Neither replaces the other.

### Commit conventions

[Conventional Commits](https://www.conventionalcommits.org/). Types in active use: `feat`, `fix`, `docs`, `refactor`, `chore`, `test`, `ci`, `perf`. Eager-claim commits use `chore(session):`. Always attribute via the `Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>` footer.

`engine/tools/validate.py` runs in the pre-commit hook. Hard-fails block the commit. Soft-warns are recorded in `engine/session/current.json`'s `outcome_summary`. See `engine/operations/tools-validate-interpretation.md`.

### Posture vs machinery

Some rules above remain postures held by AI discipline:

- The **pushback rule** — no log, no audit; a session that fails to surface a real risk leaves no trace.
- The **startup ceremony order** — steps 1, 3, and 4 run by judgment; nothing prevents skipping or reordering. Step 2 (the cadence trigger) is now mechanized via the `SessionStart` hook per [ADR 0043](engine/adr/0043-hook-architecture.md), but the order in which the AI consumes the surface remains posture.
- **Skill invocation** — the three build-mode Skills (`session-build-lifecycle`, `session-shutdown-sequence`, `build-readiness-gate`) carry `disable-model-invocation: true`. The AI is responsible for invoking each at the appropriate moment; the harness will not auto-fire them.
- **Mid-session context-pressure monitoring** (closed at S-0048 per [Issue #4](https://github.com/StarshipSuperjam/paideia/issues/4) as intractable) — the 60% / 70% / 80% budget percentages in the **Budget guidance** section above are AI posture, not machinery, and remain so by deliberate choice. Live transcript-token measurement is infrastructurally blocked: Claude Code does not expose the in-progress transcript path (or a session-id env var that would let a hook resolve the transcript) to hook subprocesses. **Operating policy**: the user manually monitors context load during routine review. The cost of the gap is captured by review, not by mechanism. (Pre-S-0083, `transcript_token_pct` was captured at shutdown for cross-session telemetry per ADR 0049 decision 3; that telemetry retired at S-0083 per Issues #21 and #32 after producing zero behavioral signal across three audits.)

These postures don't admit clean mechanical detection. The cost is that drift accumulates silently if discipline lapses. The mitigation is awareness — a session that knows the rule is unenforced can hold itself accountable rather than mistake the absence of an alarm for the presence of compliance.

Pre-S-0078 the deliberate uses of MemPalace (boot `mempalace_search`, boot `mempalace_diary_read`, shutdown `mempalace_diary_write`, manual `pushback`/`lesson` capture) were posture-only — Issue [#27](https://github.com/StarshipSuperjam/paideia/issues/27) confirmed 12 of 16 Phase 5 routine sessions silently skipped both the diary write AND the `diary_skipped` self-record, leaving the persistent-warn 3-of-5 surface inert in both directions. The S-0078 fix is end-to-end: a `PostToolUse` hook (`engine/tools/hooks/post-mempalace-tool-use.sh`) appends one JSONL line per `mcp__mempalace__*` call to `engine/session/current_mempalace.jsonl`; `engine/tools/scan_mempalace_activity.py` rolls the JSONL up into a structured `mempalace_activity` field on `engine/session/current.json` at shutdown step 0; `validate.py --final-check` (gated CLI flag, only invoked by the shutdown sequence) reads `mempalace_activity` and emits two soft-warns (`mempalace_boot_query_skipped`, `mempalace_diary_read_skipped`) and one hard-fail (`mempalace_diary_write_skipped`) keyed off zero call-counts. The hard-fail carries an explicit acknowledgement-token escape hatch (write `mempalace_unavailable_acknowledged: <reason>` into `outcome_summary` to downgrade to `mempalace_diary_write_acknowledged_skip`); persistent acknowledged-skips fire the same 3-of-5 escalation as unacknowledged ones. The pre-S-0078 `diary_skipped` field was renamed to `mempalace_diary_write_skipped` across existing archives via `engine/tools/migrate_diary_skipped_archive_field.py`. The companion fix to the `audit_archive_structured_fields.py` parameterization (closes Issue [#20](https://github.com/StarshipSuperjam/paideia/issues/20)) declares the new `mempalace_activity` field in `REQUIRED_ARCHIVE_FIELDS` so future structured-field ADRs add a row, not a code change. Closes Issues #27 and #20.
- **Build-mode lifecycle pushes via wrapper tool + close-side worktree preservation + boot-time bulk sweep** are mechanized per [ADR 0076](engine/adr/0076-build-mode-lifecycle-push-wrapping.md) (S-0138) and its S-0143 Amendment v2. [`engine/tools/build_lifecycle_push.py`](engine/tools/build_lifecycle_push.py) is the sibling to `routine_lifecycle_push.py` per ADR 0054 — same subprocess-bypass pattern, same eager-claim + close shape verifiers (imported byte-identical from the routine wrapper), but a deliberately looser deliverable verifier because build sessions have no machine-readable scope contract (the user-approved plan is the scope-of-record, in the plan file + conversation). Required because the harness "Default Branch Push" gate denies raw `git push origin main` from interactive build sessions despite `/start-engine` authorization — falsifying ADR 0054's "interactive sessions don't trigger the gate" out-of-scope assertion (S-0137 empirical record). **Close-side worktree preservation** (S-0143 Amendment v2): the closing session's worktree is NOT swept at close. Both [`engine/tools/routine_worktree_sweep.py`](engine/tools/routine_worktree_sweep.py) and [`engine/tools/sweep_worktrees.sh`](engine/tools/sweep_worktrees.sh) carry a `caller's-own-worktree` pre-flight that refuses sweep when the resolved target equals the caller's CWD; the per-worktree tool emits a structured preserve-report (path + branch + merged/ahead/behind + dirty files + last commit + actionable guidance) on refusal. The pre-S-0143 wiring (post-close `routine_worktree_sweep.py` invocation at S-0142) was reversed at S-0143 after S-0142's own close empirically wiped the user's working folder before they could follow up. Accumulated prior-session worktrees are reaped at the **next session's boot** by [`engine/tools/hooks/session-start.sh`](engine/tools/hooks/session-start.sh) invoking `sweep_worktrees.sh --apply --quiet` (gated on `CONFLICT_STATUS == no-conflict` to avoid sibling-session races; conservative pre-flight ensures only `claude/*` + clean + merged worktrees are reaped). Opt-in escape-hatch `--allow-caller-self` for tests and manual recovery; production close ceremonies must not set it. **Per-worktree liveness marker** (S-0157 amendment, [Issue #120](https://github.com/StarshipSuperjam/paideia/issues/120)): `session-start.sh` writes a `session-live` marker into each worktree's private git dir at every boot — *before* eager-claim, so even a plan/exploration session that never claims a slot is protected; both sweep tools preserve any worktree whose marker is within a 24h freshness window. This closes the bystander gap the caller-self check does not cover — a pre-eager-claim session's worktree is `claude/*` + clean + merged (no commits yet) and would otherwise be reaped by a sibling's boot-time sweep. Layer 1 doc: [`engine/operations/session-shutdown-sequence.md`](engine/operations/session-shutdown-sequence.md) step 11.
- **Routine-mode wedge detect-and-pause** is mechanized per [ADR 0060](engine/adr/0060-routine-wedge-detect-and-pause.md) (S-0118). Closes Issue #58 (the S-0117 wedge spawned hourly HANDOFF appends because no boot step inspected for the halted-routine shape). [`engine/tools/routine_wedge_detect.py`](engine/tools/routine_wedge_detect.py) runs at routine boot step 0c (between concurrency-lock acquire per ADR 0082 and target precondition), inspects `register_state.json` + `current.json` + `auto_target.json` + HEAD-vs-`origin/main` for the wedge shape (register/current/task all `in_progress`, no archive at HEAD, lock acquirable, HEAD at-or-behind origin), and on detection idempotently authors a single GitHub Issue + a single HANDOFF section + commits and pushes via subprocess (same bypass pattern as `routine_lifecycle_push.py` per ADR 0054). Exit codes 0 (no wedge, proceed) / 2 (wedge handled, exit cleanly without claiming) / 3 (ambiguous shape, HANDOFF + exit) / 5 (generic failure, HANDOFF + exit). The auto-administrative-close alternative was rejected because synthesizing structured archive fields (`outcome_summary`, `outcome_summary_soft_warns`, `mempalace_activity`) without the halted session's input silently corrupts downstream telemetry per ADR 0042 + ADR 0049 + ADR 0056. 14 pytests cover predicate conjuncts + idempotency + dry-run no-mutate. First-exercise readiness note at [`engine/build_readiness/routine_wedge_detect_first_exercise.md`](engine/build_readiness/routine_wedge_detect_first_exercise.md) per [ADR 0053](engine/adr/0053-mechanism-first-exercise-gate.md) (T1-A — first natural production wedge — closes when next halted-routine wedge naturally occurs).
- **Skill ↔ Layer-1 procedure-enumeration parity** is mechanized per [ADR 0089](engine/adr/0089-skill-layer1-parity-validator-check.md) (S-0163; closes Issue #129). [`engine/tools/validate.py`](engine/tools/validate.py)'s structural-phase `validate_skill_layer1_parity()` / `skill_layer1_parity_drift` soft-warn compares each of the four recipe Skills' procedure step-number *set* against its Layer-1 ops doc's (config: the `_SKILL_LAYER1_PAIRS` tuple). It catches the enumeration-drift class the #122-#129 cluster surfaced — a step present on one side and absent from the other — and falsifies ADR 0044's "drift flows doc → skill only" assumption (drift ran skill↔skill at #125, command↔skill at #123). Compares step numbers, not titles: skill voice and reference voice legitimately differ in wording per ADR 0044. **Intra-step content parity remains posture** — a step's prose drifting from its sibling's, or a step omitting required fields (#122, #126 part 1), is not caught by a step-set comparison; a structural check is the wrong tool for semantic content parity (see ADR 0089 Alternatives Considered). 15 pytests in `test_validate_skill_parity.py`.
- **Routine-mode permission posture** (revised at S-0048 per Issue #6): routines run in **Auto** permission mode; the harness allowlist is supplementary and serves *interactive* sessions only. The S-0046 attempt to author a comprehensive Bash allowlist for routines didn't generalize — the AI inside a routine session naturally uses compound Bash forms (`cmd1; cmd2`, `cmd && cmd`) and the harness matches the *whole* compound command against patterns rather than the leading prefix, so pattern proliferation never converges. Auto mode eliminates the brittle compound-form mismatch. Project-level mechanical safeguards remain load-bearing: `check_routine_scope.py --plan` at boot, `check_routine_scope.py --staged` at every commit, master-plan-integrity check on `auto_target.json`, `audit_handoff_dispositions.py` at shutdown, `validate.py` soft-warns (`routine_no_target_reference`, `routine_issue_spam`), and the eager-claim atomic protocol — none of these depend on harness permissions. The routine-mode-lifecycle Skill body explicitly refuses destructive operations (`rm -rf`, `git reset --hard`, force-push) and writes HANDOFF instead, covering the destructive-op gap that no project tool can mechanically detect at staging time. The S-0046 same-commit pairing rule for routine-procedure changes is retired; allowlist additions are still good practice for interactive-session ergonomics but no longer load-bearing for routines. The S-0046 allowlist additions remain in `.claude/settings.json` for interactive use. Layer 1 doc: [`engine/operations/routine-mode-operations.md`](engine/operations/routine-mode-operations.md) "Harness permission allowlist".

Soft enforcement preserves AI judgment. A reminder can be acknowledged and overridden in legitimate cases (an ADR supersession that intentionally reuses an existing decision drawer; a STATE.md edit with a transient placeholder mid-edit). Hard enforcement was rejected for these rules in [ADR 0043](engine/adr/0043-hook-architecture.md) because the legitimate-exception surface is non-empty.
